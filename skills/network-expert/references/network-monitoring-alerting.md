# 网络运维 - 监控与告警系统

## 一、网络运维概述

### 定义
网络运维是指对网络基础设施进行监控、维护、优化和故障处理的系统性工作，确保网络稳定、高效、安全运行。

### 核心目标
- **可用性**：网络服务持续可用
- **性能**：网络性能达标
- **安全**：网络安全无事故
- **效率**：运维效率最大化

## 二、监控体系架构

### 1. 监控层次
```
┌─────────────────────────────────────┐
│           业务监控                   │
│    （业务指标、用户体验）              │
├─────────────────────────────────────┤
│           应用监控                   │
│    （应用性能、响应时间）              │
├─────────────────────────────────────┤
│           系统监控                   │
│    （CPU、内存、磁盘、进程）           │
├─────────────────────────────────────┤
│           网络监控                   │
│    （流量、延迟、丢包、连接）          │
├─────────────────────────────────────┤
│           基础设施监控                │
│    （服务器、交换机、路由器）          │
└─────────────────────────────────────┘
```

### 2. 监控指标分类

#### 网络层指标
| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 带宽利用率 | 链路带宽占用率 | >80% |
| 丢包率 | 数据包丢失比例 | >0.1% |
| 延迟 | 网络往返时间 | >100ms |
| 抖动 | 延迟波动 | >50ms |
| 连接数 | 活动连接数 | >80%最大值 |

#### 设备层指标
| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| CPU使用率 | 设备CPU占用 | >80% |
| 内存使用率 | 设备内存占用 | >85% |
| 接口状态 | 端口up/down | down告警 |
| 温度 | 设备温度 | >70℃ |
| 电源状态 | 电源工作状态 | 异常告警 |

#### 应用层指标
| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 响应时间 | 请求响应耗时 | >2s |
| 错误率 | 请求失败比例 | >1% |
| 并发数 | 同时在线用户 | >90%容量 |
| 队列长度 | 等待请求数 | >100 |

## 四、告警系统设计

### 1. 告警级别
| 级别 | 名称 | 处理时间 | 示例 |
|------|------|----------|------|
| P0 | 紧急 | 5分钟 | 核心业务中断 |
| P1 | 严重 | 15分钟 | 关键功能降级 |
| P2 | 警告 | 1小时 | 性能下降 |
| P3 | 提示 | 24小时 | 非关键告警 |

### 2. 告警规则设计

#### 单指标告警
```yaml
# CPU使用率告警
name: cpu_usage_alert
condition: cpu_usage > 80
duration: 5m
severity: warning
action: notify_oncall
```

#### 复合条件告警
```yaml
# 网络拥塞告警
name: network_congestion
conditions:
  - bandwidth_usage > 70%
  - packet_loss > 0.05%
  - latency > 50ms
logic: all
severity: critical
action: page_network_team
```

#### 趋势告警
```yaml
# 流量异常增长
name: traffic_anomaly
condition: traffic_growth_rate > 200%
baseline: same_time_last_week
severity: warning
action: notify_security
```

### 3. 告警收敛与降噪

#### 告警去重
- 相同告警5分钟内只告警一次
- 同一设备的关联告警合并

#### 告警抑制
- 上游设备故障，抑制下游设备告警
- 维护窗口期间，抑制非紧急告警

#### 告警静默
- 计划维护期间自动静默
- 已知问题的临时静默

## 五、故障处理流程

### 1. 故障分级
| 级别 | 定义 | 影响 | 响应时间 |
|------|------|------|----------|
| P0 | 核心业务完全中断 | 全部用户受影响 | <5分钟 |
| P1 | 核心功能降级 | 大部分用户受影响 | <15分钟 |
| P2 | 非核心功能异常 | 部分用户受影响 | <1小时 |
| P3 | 性能轻微下降 | 个别用户受影响 | <4小时 |

### 2. 故障处理SOP

#### 故障发现
```
监控告警 → 确认告警 → 初步判断
```

#### 故障响应
```
P0/P1: 立即响应，召集相关人员
P2/P3: 工单流转，按优先级处理
```

#### 故障处理
```
止损优先 → 根因分析 → 修复验证
```

#### 故障复盘
```
时间线梳理 → 根因定位 → 改进措施 → 文档归档
```

### 3. 故障止损策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 回滚 | 恢复上一版本 | 发布后故障 |
| 降级 | 关闭非核心功能 | 性能瓶颈 |
| 熔断 | 自动切断故障点 | 服务异常 |
| 限流 | 控制请求速率 | 流量突增 |
| 熔断器 | 自动跳过故障服务 | 依赖服务异常 |

## 七、SLA/SLO/SLI管理

### 1. 概念定义

#### SLA（服务等级协议）
- 对用户的承诺
- 具有法律约束力
- 示例：99.9%可用性

#### SLO（服务等级目标）
- 内部目标
- 量化可衡量
- 示例：99.95%可用性

#### SLI（服务等级指标）
- 具体测量值
- 客观可计算
- 示例：成功请求数/总请求数

### 2. 可用性计算

#### 公式
```
可用性 = (总时间 - 故障时间) / 总时间 × 100%
```

#### SLA等级
| 等级 | 可用性 | 年故障时间 | 适用场景 |
|------|--------|-----------|----------|
| 99% | 99% | 87.6小时 | 非关键业务 |
| 99.9% | 99.9% | 8.76小时 | 一般业务 |
| 99.99% | 99.99% | 52.6分钟 | 重要业务 |
| 99.999% | 99.999% | 5.26分钟 | 核心业务 |

### 3. 错误预算

#### 概念
```
错误预算 = 1 - SLO
```

#### 使用方式
- 正常消耗：允许一定的故障
- 预算耗尽：停止新功能发布
- 预算剩余：可进行创新尝试

## 八、常见网络问题排查

### 1. 连通性问题

#### 排查步骤
```
ping → traceroute → 检查路由 → 检查防火墙
```

#### 常见原因
| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| ping不通 | 物理连接、路由、ACL | 检查线路、路由表 |
| 部分可通 | 路由不对称、NAT问题 | 检查路由、NAT配置 |
| 间歇断开 | 链路抖动、拥塞 | 检查链路质量 |

### 2. 性能问题

#### 排查步骤
```
带宽测试 → 延迟测量 → 丢包检测 → 检查QoS
```

#### 常见原因
| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 延迟高 | 拥塞、路由绕行 | 扩容、优化路由 |
| 丢包 | 链路质量、拥塞 | 检查链路、流量控制 |
| 抖动大 | 队列延迟、负载不均 | QoS调度优化 |

### 3. DNS问题

#### 排查步骤
```
nslookup → dig → 检查DNS配置 → 检查DNS服务器
```

#### 常见原因
| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 解析失败 | DNS服务器故障 | 配置备用DNS |
| 解析慢 | DNS服务器响应慢 | 本地缓存 |
| 解析错误 | 缓存污染 | 清除缓存 |

## 九、运维最佳实践

### 1. 监控设计原则
- **全覆盖**：不留监控盲区
- **分级**：根据重要程度分级
- **可观测**：指标、日志、追踪三位一体

### 2. 告警设计原则
- **少而精**：减少无效告警
- **可操作**：每个告警都有处理方法
- **及时性**：重要告警秒级通知

### 3. 故障处理原则
- **止损优先**：先恢复服务，再找根因
- **快速响应**：建立值班机制
- **事后复盘**：每次故障都要复盘

### 4. 持续改进
- **定期巡检**：主动发现问题
- **容量规划**：提前扩容
- **演练测试**：故障演练、灾备切换
