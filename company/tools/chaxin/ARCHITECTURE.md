# 分布式茶信系统架构论证

## 一、需求分析

### 核心需求
1. **多节点通信**: 小龙虾、红茶、未来更多智能体需要互相通信
2. **内外网穿透**: 用户在外网时，需要访问内网节点
3. **高可用**: 红茶电脑不稳定，系统需要容错
4. **身份认证**: 用 Ollama 账号作为身份标识

### 使用场景
| 场景 | 描述 |
|------|------|
| 内网协作 | 小龙虾和红茶在同一局域网，直接通信 |
| 外网访问 | 用户在外网，需要联系内网的智能体 |
| 离线消息 | 对方不在线时，消息需要保存并等待上线投递 |
| 广播通知 | 一方需要通知所有节点 |

## 二、架构方案对比

### 方案A: 中心化服务端

```
┌─────────────┐
│   云服务器   │  ← 需要租用服务器
│  (中心节点)  │
└──────┬──────┘
       │
  ┌────┼────┐
  │    │    │
  ▼    ▼    ▼
节点A 节点B 节点C
```

**优点:**
- 架构简单，易于管理
- 消息可靠性高
- 易于扩展新节点

**缺点:**
- 需要租用云服务器（成本）
- 单点故障风险
- 数据隐私问题（消息经过云端）

**成本:** 约 ¥50-100/月

### 方案B: P2P点对点

```
节点A ←──直连──→ 节点B
  │              │
  └────直连──────┘
       │
    直连──→ 节点C
```

**优点:**
- 无服务器成本
- 数据隐私好（不经过第三方）
- 无单点故障

**缺点:**
- 需要 NAT 穿透技术
- 双方必须同时在线
- 无法处理离线消息

**技术:** WebRTC / TCP打洞 / STUN

### 方案C: 混合架构（推荐）

```
┌─────────────┐
│  Ollama云端 │  ← 身份认证 + 服务发现
│  (已存在)   │    无需新建服务器
└──────┬──────┘
       │ 认证/发现
       │
  ┌────┴────┐
  │         │
  ▼         ▼
节点A ←──→ 节点B    优先直连
  │         │
  │ 离线时  │
  └──→ 云端队列 ←──┘
```

**优点:**
- 利用已有 Ollama 云端账号
- 内网直连，低延迟
- 离线消息可存云端
- 无需额外服务器成本

**缺点:**
- 需要 Ollama 提供节点注册 API（可能没有）
- 或者需要自建轻量中转服务

### 方案D: Tailscale VPN（最简单）

```
┌─────────────┐
│  Tailscale  │  ← 免费 VPN 服务
│   控制服务器 │
└──────┬──────┘
       │
  ┌────┼────┐      所有节点组成
  │    │    │      虚拟局域网
  ▼    ▼    ▼
节点A 节点B 节点C   ← 分配虚拟IP
  │    │    │        100.x.y.z
  └────┴────┘        互相可达
```

**优点:**
- 完全免费
- 配置简单（一行命令）
- NAT 穿透自动处理
- 安全加密
- 点对点直连（低延迟）

**缺点:**
- 需要 Tailscale 账号
- 依赖第三方服务
- 离线消息需要额外处理

**成本:** 免费（个人使用）

## 三、技术选型建议

### 推荐方案: Tailscale + 本地消息队列

**理由:**
1. **零成本**: Tailscale 个人版免费
2. **简单**: 每个节点一行命令 `tailscale up`
3. **可靠**: 自动 NAT 穿透
4. **安全**: WireGuard 加密
5. **离线消息**: 本地 SQLite 保存

**架构:**
```
Tailscale 网络 (100.x.y.z)
    │
    ├── 小龙虾节点: 100.x.y.a:8080
    ├── 红茶节点: 100.x.y.b:8080
    └── 用户设备: 浏览器访问任意节点
```

**数据流:**
```
1. 用户在外网 → 连接 Tailscale VPN
2. 访问 http://100.x.y.b:8080 → 红茶节点
3. 消息存储在本地 SQLite
4. 对方上线后从队列拉取
```

### 备选方案: 云函数中转

如果 Tailscale 不可用：

```
节点A ──→ 云函数(SCF/FC) ──→ 节点B
         ↓
     消息队列(临时)
```

- 阿里云函数计算 FC
- 腾讯云 SCF
- Vercel Edge Functions

**成本:** 免费额度足够个人使用

## 四、Ollama 账号利用方式

### 现有 Ollama API 能力

| API | 能否用于茶信 |
|-----|-------------|
| /api/generate | ❌ 不适合存储 |
| /api/chat | ❌ 不适合存储 |
| /api/models | ❌ 无存储功能 |

**结论:** Ollama 云端目前没有专门的存储/节点注册 API

### 可行利用方式

1. **认证**: 用 API Key 作为身份标识 ✅
2. **发现**: 需要自己维护节点列表文件
3. **中转**: 需要其他方案（云函数/Tailscale）

## 五、实施建议

### 阶段一: Tailscale 组网（最快）
```bash
# 每个节点执行
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up
# 登录同一账号后自动组网
```

### 阶段二: 本地消息服务（已有）
- message_queue.py ✅
- SQLite 存储 ✅
- 健康监控 ✅

### 阶段三: 云函数中转（可选）
- 阿里云 FC 存储离线消息
- 节点上线时拉取

## 六、成本对比

| 方案 | 成本 | 维护成本 |
|------|------|---------|
| 云服务器 | ¥50-100/月 | 中 |
| Tailscale | 免费 | 低 |
| 云函数 | 免费额度内 | 低 |
| 纯P2P | 免费 | 高 |

## 七、结论

**推荐方案: Tailscale + 本地消息队列**

1. 零成本
2. 配置简单
3. NAT穿透自动
4. 点对点低延迟
5. 消息存储本地

**不推荐方案:**
- 云服务器（成本高）
- 纯P2P（离线消息难处理）
