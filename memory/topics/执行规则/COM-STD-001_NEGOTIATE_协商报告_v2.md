## 一、ECS小智 🧠 完整意见

### 1. 可行性评估

| 方式 | 认可 | 本节点可行 | 说明 |
|------|:----:|:---------:|------|
| ① SSH 直连 | ✅ | ✅ | 密钥对 + 安全组白名单 |
| ② Task API | ✅ | ✅ | 需安装 agent + token |
| ③ Gateway HTTP API | ✅ | ✅ | 16ms 延迟意味着需同区域 |
| **④ WS (18789)** | ⚠️ 部分 | ❌ | **ECS不跑OpenClaw，此通道不可用** |
| ⑤ proot→agent | ✅ | ✅ | 核心方案，建议 mTLS 加密 |
| ⑥ Broadcast | ✅ | ✅ | 建议走Redis Channel |
| **⑦ WMI** | ✅ | ❌ | **Linux节点不支持** |

### 2. 开放/关闭建议

**✅ 开放**: ①②③⑤⑥ — 均需鉴权和内网绑定
**❌ 关闭**: ④⑦ — ECS环境不支持
**🔒 安全补充**: 所有HTTP/WS接口应绑内网IP + JWT/Token鉴权；SSH密钥定期轮换

### 3. 补充建议

**新增方式⑧ gRPC 双向流** — 实时状态同步、AI推理流，比HTTP REST更高效
**新增方式⑨ 共享存储信号** — NFS共享目录 + inotify触发消息，轻量零依赖
**新增方式⑩ RDMA/InfiniBand** — 极低延迟场景，列为可选高级方式

**改进方向⑤**: proot→agent 应明确定义心跳重连和滚动升级策略
**改进方向③**: 16ms 应说明 P99 基准，引入排队降级
**改进方向②**: Task API 应支持异步回调

## 二、Worker-Arm 🤖 完整意见

### 1. 可行性评估

| 方式 | 本节点可行 | 说明 |
|:----:|:---------:|------|
| ① SSH | ✅ | 密钥认证已启用 |
| ② Task API | ✅ | ComputeHub Agent 监听中 |
| ③ HTTP API | ✅ | 局域网延迟1-3ms |
| ④ WS (18789) | ⚠️ 有限 | **仅同节点AI对话，不能跨节点** |
| ⑤ proot→agent | ✅ | 实测通过，延迟20-30ms |
| ⑥ Broadcast | ✅ | 基于 Redis Pub/Sub |
| ⑦ WMI | ❌ | ARM Linux → Windows专属 |

### 2. 开放/关闭建议

| 通道 | 意愿 | 条件 |
|:----:|:----:|------|
| ① SSH | ✅ 开放 | 仅管理员网段 + 密钥 + 审计日志 |
| ② Task API | ✅ 开放 | 建议命令沙箱 + 白名单 |
| ③ HTTP API | ✅ 开放 | 仅内网，不暴露公网 |
| ④ WS | ✅ 开放 | 仅本地localhost |
| ⑤ proot→agent | ✅ 开放 | 核心方案，保持畅通 |
| ⑥ Broadcast | ✅ 开放 | 仅接收授权节点广播 |
| ⑦ WMI | ❌ 关闭 | 本地防火墙禁用135/445/5985 |

### 3. 补充建议

**① 新增 gRPC 双向流** — 实时数据交换（传感器、日志、中间状态），10ms内延迟
**② 新增文件传输标准** — SCP/SFTP 或 HTTP PUT/GET 上传下载，集群共享目录
**③ 增加服务发现与健康检测** — 节点注册到Consul或内置Register API，避免IP硬编码
**④ 统一 mTLS 加密** — 所有跨节点通信启用TLS 1.3 + 集群CA证书双向认证
