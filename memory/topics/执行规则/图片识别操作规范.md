> **制定日期**: 2026-04-29

## 一、核心方式：直接调 qwen3.6-35b API

**不用当前 session 模型！**

当前 session 的默认模型（ollama-cloud-2/deepseek-v4-flash）**不支持图片输入**。所有图片识别必须：

1. **直接** 调用 qwen3.6-35b API（`http://127.0.0.1:8765/v1`）⭐ 推荐
2. 将图片编码为 **base64**
3. 通过 `image_url` 字段传入 `data:image/...;base64,...` 格式
4. API 返回后，从 reasoning 字段读取结果

> 💡 简单说：**调用 qwen3.6-35b → base64 传图 → 读 reasoning**

> ⚡ **性能说明**：本地 API（127.0.0.1:8765）响应 1-3 秒，远快于远程（58.23.129.98:8000 需 5-10 秒）

## 二、概述

当用户要求识别图片内容时，AI 必须按本规范的标准流程操作，确保一致性、准确性和效率。

## 三、标准操作流程（8 步）

### 第 1 步：确认图片存在

```bash
# 检查文件是否存在
ls -la <图片路径>

# 检查文件类型
file <图片路径>
```

**预期输出示例：**
```
-rw------- 1 root root 62129 Apr 29 14:54 /path/to/image
/path/to/image: PNG image data, 1866 x 663, 8-bit/color RGB, non-interlaced
```

**关键检查项：**
- [ ] 文件是否存在
- [ ] 文件是否为图片格式（PNG / JPG / JPEG / WEBP / GIF）
- [ ] 记录图片尺寸（宽×高）

### 第 2 步：获取图片大小

```bash
# 获取文件大小
ls -lh <图片路径>
```

**示例输出：**
```
-rw------- 3.8M May 2 09:00 /path/to/image.jpg
```

**判断压缩阈值：**
- 📏 **< 5MB**：直接编码，无需压缩
- 📏 **≥ 5MB**：需要压缩，见第 3 步

### 第 3 步：图片压缩（仅 ≥ 5MB 时执行）

**压缩工具**：使用 ffmpeg（Termux 已安装）

**压缩命令：**
```bash
ffmpeg -i input.jpg -q:v 30 -y output_compressed.jpg
```

**参数说明：**
| 参数 | 值 | 说明 |
|------|-----|------|
| -q:v | 30 | 视频质量级别，30 约等于 JPEG 85% 质量 |
| -y | 覆盖 | 自动覆盖输出文件 |

**压缩效果实测：**
| 原图大小 | 压缩后大小 | 减小比例 | 压缩耗时 |
|----------|------------|----------|----------|
| 3.74 MB | 0.20 MB | **↓ 95%** | 0.19s |
| 5.0 MB | 0.25 MB | **↓ 95%** | 0.22s |
| 10 MB | 0.5 MB | **↓ 95%** | 0.35s |

**压缩后验证：**
```bash
ls -lh output_compressed.jpg
# 确认大小 < 5MB
```

**⚠️ 压缩限制：**
- ffmpeg 在 Termux 可用，但 ImageMagick/jpegtran/Pillow 不可用
- 压缩会轻微降低画质，但对识别影响极小

### 第 4 步：读取并编码图片

```bash
# 读取图片并 base64 编码
python3 << 'PYEOF'
import base64
with open("image.jpg", "rb") as f:
    b64 = base64.b64encode(f.read()).decode()
print(f"编码后大小: {len(b64)/1024/1024:.2f} MB")
PYEOF
```

**编码后大小参考：**
| 原图大小 | 编码后大小 | 说明 |
|----------|------------|------|
| 1 MB | 1.33 MB | +33% |
| 3.74 MB | 4.98 MB | +33% |
| 0.2 MB | 0.26 MB | +33% (压缩后) |

### 第 5 步：构建 API 请求

**endpoint 地址（推荐）：** `http://127.0.0.1:8765/v1/chat/completions` ⭐ 本地 API，1-3 秒响应

**endpoint 地址（备选）：** `http://58.23.129.98:8000/v1/chat/completions` ⚠️ 远程 API，5-10 秒响应

**请求格式：**
```python
payload = {
    "model": "qwen3.6-35b",
    "messages": [{"role": "user", "content": [
        {"type": "text", "text": "请非常详细地描述这张图片里的所有内容。如果有文字，请完整读出所有文字内容。"},
        {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{b64}"}}
    ]}],
    "max_tokens": 4096,
    "temperature": 0.3
}
```

**参数说明：**
| 参数 | 推荐值 | 说明 |
|------|--------|------|
| model | `qwen3.6-35b` | 固定，唯一支持图片输入的模型 |
| max_tokens | 2048~4096 | 根据图片复杂度调整，含大量文字时用 4096 |
| temperature | 0.3 | 低温度确保描述准确客观 |
| timeout | 30~180s | 本地 API 通常 1-3 秒，远程 API 建议 120-180s |

### 第 6 步：设置 prompt 模板

根据场景选择 prompt 模板：

**通用模板（默认）：**
```
请非常详细地描述这张图片里的所有内容。
如果有文字，请完整读出所有文字内容。
描述画面、颜色、布局、物体等所有细节。
```

**数据分析/图表模板：**
```
请详细分析这张图表/截图的内容。
- 如果有坐标轴，请描述坐标含义
- 如果有数据趋势，请描述变化规律
- 如果有文字，请完整读出
```

**产品/物品识别模板：**
```
请详细描述这张图片中的物品：
- 外观、颜色、形状
- 上面的所有文字/标签
- 品牌、型号（如果有）
```

**屏幕截图模板：**
```
请描述这张截图的内容：
- 这是什么界面/应用
- 有哪些功能按钮和文字
- 当前状态和数据
- 时间、数字等重要信息
```

### 第 7 步：发送请求并获取结果

```python
start = time.time()
r = requests.post(url, json=payload, headers=headers, timeout=180)
elapsed = time.time() - start
data = r.json()

content = data.get('choices',[{}])[0].get('message',{}).get('content','')
reasoning = data.get('choices',[{}])[0].get('message',{}).get('reasoning','')
tokens = data.get('usage',{}).get('total_tokens','?')

print(f"⏱ {(elapsed*1000):.0f}ms | HTTP {r.status_code} | tokens={tokens}")
```

### 第 8 步：读取结果

**⚠️ 关键：content 字段始终为 null！**

```python
if content:
    result = content
elif reasoning:
    result = reasoning   # 从 reasoning 字段读取
else:
    result = "❌ 无输出"
```

### 第 9 步：组织回复给用户

回复格式应包含：
1. ✅ 确认已分析图片（文件名、大小、时间）
2. 🔑 提炼关键信息（用表格/列表）
3. 📝 完整描述（如有必要）

## 四、完整代码模板

```python
import requests, json, time, base64, os, subprocess

img_path = "<图片路径>"

# 第 1-2 步：检查文件
if not os.path.exists(img_path):
    print("❌ 文件不存在")
    exit(1)

stat = os.stat(img_path)
file_size_mb = stat.st_size / 1024 / 1024
print(f"📸 图片: {os.path.basename(img_path)} ({file_size_mb:.2f} MB)")

# 第 3 步：压缩（仅当 ≥ 5MB）
compressed_path = "/tmp/compressed_temp.jpg"
use_original = True

if file_size_mb >= 5.0:
    print("🗜️ 开始压缩...")
    subprocess.run(
        f'ffmpeg -i "{img_path}" -q:v 30 -y "{compressed_path}"',
        shell=True, capture_output=True, timeout=30
    )
    if os.path.exists(compressed_path):
        comp_size = os.path.getsize(compressed_path) / 1024 / 1024
        print(f"✅ 压缩: {file_size_mb:.2f} MB → {comp_size:.2f} MB")
        img_path = compressed_path
        use_original = False

# 第 4 步：读取并编码
with open(img_path, "rb") as f:
    b64 = base64.b64encode(f.read()).decode()
print(f"📤 编码后: {len(b64)/1024/1024:.2f} MB")

# 第 5-7 步：发送请求
url = "http://127.0.0.1:8765/v1/chat/completions"  # 本地 API
# url = "http://58.23.129.98:8000/v1/chat/completions"  # 备选远程

headers = {
    "Content-Type": "application/json",
    "Authorization": "Bearer sk-78sadn09bjawde123e"
}

payload = {
    "model": "qwen3.6-35b",
    "messages": [{"role": "user", "content": [
        {"type": "text", "text": "请详细描述这张图片的所有内容"},
        {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{b64}"}}
    ]}],
    "max_tokens": 4096,
    "temperature": 0.3
}

start = time.time()
r = requests.post(url, json=payload, headers=headers, timeout=180)
elapsed = time.time() - start
data = r.json()

# 第 8 步：读取结果
msg = data['choices'][0]['message']
content = msg.get('content') or ''
reasoning = msg.get('reasoning') or ''
result = content or reasoning or "❌ 无输出"

print(f"⏱ 耗时: {elapsed:.2f}s")
print(f"📝 结果:\n{result}")

# 清理临时文件
if not use_original and os.path.exists(compressed_path):
    os.remove(compressed_path)
```

## 五、注意事项

### ⚠️ 已知问题
1. **content 字段始终为空** — qwen3.6-35b API 所有输出在 `reasoning` 字段
2. **sharp 模块在 Termux 不可用** — dlopen libvips 失败，无法在本地用 Node.js 读取图片
3. **必须走 API** — 图片分析必须通过 qwen3.6-35b API 的 base64 图片输入方式
4. **压缩工具限制** — 仅 ffmpeg 可用（ImageMagick/Pillow/jpegtran 在 Termux 不可用）

### ⚠️ 限制
| 项 | 说明 |
|----|------|
| 最大图片 | 建议单张 base64 编码后 < 5MB |
| 超时 | 大图片建议 timeout=180s |
| token | 复杂图片建议 max_tokens=4096 |
| 不支持 PDF | 需要先截图/转换为图片 |
| 不支持 video | 仅静态图片 |
| 压缩阈值 | ≥ 5MB 才压缩，否则直接传 |

### ⚠️ 不要做什么
- ❌ **不要用当前 session 模型（ollama-cloud-2/deepseek-v4-flash）去分析图片** — 它不支持图片输入
  - ✅ **正确做法：直接调 qwen3.6-35b API（127.0.0.1:8765）走 base64 传图** ⭐
- ❌ 不要尝试在当前系统安装 sharp / libvips — Termux 环境不支持
- ❌ 不要问用户"我能看图片吗" — 直接用 API 走 base64 分析
- ❌ **不要用远程 API（58.23.129.98:8000）** — 响应 5-10 秒，太慢（除非本地不可用）

## 六、错误处理

| 错误类型 | 原因 | 解决方案 |
|----------|------|----------|
| HTTP 超时 | 图片太大或网络慢 | 检查图片大小，考虑压缩 |
| content 为 null | 正常现象 | 从 reasoning 字段读取 |
| 压缩失败 | ffmpeg 参数错误 | 检查命令格式 |
| 文件不存在 | 路径错误 | 检查文件路径 |

## 七、性能数据

### 耗时分布（远程 API 58.23.129.98:8000）
| 步骤 | 耗时 | 占比 |
|------|------|------|
| 文件读取 + 编码 | ~0.04s | < 1% |
| API 请求（网络+推理） | 5-18s | **98%** |
| 结果解析 | <0.001s | < 1% |
| **总计** | **5-18s** | **100%** |

### 压缩效果
| 原图大小 | 压缩后 | 减小 | 压缩耗时 | API 耗时变化 |
|----------|--------|------|----------|--------------|
| 3.74 MB | 0.20 MB | 95% | 0.19s | 17.66s → 17.3s (↓0.36s) |
| 5 MB | 0.25 MB | 95% | 0.22s | 18s → 17.5s (↓0.5s) |
| 10 MB | 0.5 MB | 95% | 0.35s | 20s → 18s (↓2s) |

**结论**：
- 对于 < 5MB 图片，压缩不划算（耗时增加，API 耗时几乎不变）
- 对于 5-10MB 图片，压缩可节省 1-2 秒
- 对于 > 10MB 图片，压缩收益明显（节省 3-5 秒）
