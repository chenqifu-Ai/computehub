## 三、标准操作流程（8 步）

### 第 1 步：确认图片存在

```bash
# 检查文件是否存在
ls -la <图片路径>

# 检查文件类型
file <图片路径>
```

**关键检查项：**
- [ ] 文件是否存在
- [ ] 文件是否为图片格式（PNG / JPG / JPEG / WEBP / GIF）
- [ ] 记录图片尺寸（宽×高）

### 第 2 步：获取图片大小

```bash
# 获取文件大小
ls -lh <图片路径>
```

**判断压缩阈值：**
- 📏 **< 5MB**：直接编码，无需压缩
- 📏 **≥ 5MB**：需要压缩，见第 3 步

### 第 3 步：图片压缩（仅 ≥ 5MB 时执行）

**压缩工具**：使用 ffmpeg（Termux 已安装）

**压缩命令：**
```bash
ffmpeg -i input.jpg -q:v 30 -y output_compressed.jpg
```

**参数说明：**
| 参数 | 值 | 说明 |
|------|-----|------|
| -q:v | 30 | 视频质量级别，30 约等于 JPEG 85% 质量 |
| -y | 覆盖 | 自动覆盖输出文件 |

**⚠️ 压缩限制**
- ffmpeg 在 Termux 可用，但 ImageMagick/jpegtran/Pillow 不可用
- 压缩会轻微降低画质，但对识别影响极小

### 第 4 步：读取并编码图片

```bash
python3 << 'PYEOF'
import base64
with open("image.jpg", "rb") as f:
    b64 = base64.b64encode(f.read()).decode()
print(f"编码后大小: {len(b64)/1024/1024:.2f} MB")
PYEOF
```

**编码后大小参考：**
| 原图大小 | 编码后大小 | 说明 |
|----------|------------|------|
| 1 MB | 1.33 MB | +33% |
| 3.74 MB | 4.98 MB | +33% |
| 0.2 MB | 0.26 MB | +33% (压缩后) |

### 第 5 步：构建 API 请求

**endpoint 地址（推荐）：** `https://ai.zhangtuokeji.top:9090/v1/chat/completions` ⭐

**请求格式：**
```python
payload = {
    "model": "qwen3.6-35b",
    "messages": [{"role": "user", "content": [
        {"type": "text", "text": "请非常详细地描述这张图片里的所有内容。如果有文字，请完整读出所有文字内容。"},
        {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{b64}"}}
    ]}],
    "max_tokens": 4096,
    "temperature": 0.3
}
```

**参数说明：**
| 参数 | 推荐值 | 说明 |
|------|--------|------|
| model | `qwen3.6-35b` | 支持图片输入的模型 |
| max_tokens | **必须 ≥ 1024**（推荐 2048~4096） | 含大量文字时用 4096 |
| temperature | 0.3 | 低温度确保描述准确客观 |
| timeout | 60~180s | 含图片输入通常 10-30s |

### 第 6 步：设置 prompt 模板

## 四、完整代码模板

```python
import requests, json, time, base64, os, subprocess

img_path = "<图片路径>"

# 第 1-2 步：检查文件
if not os.path.exists(img_path):
    print("❌ 文件不存在")
    exit(1)

stat = os.stat(img_path)
file_size_mb = stat.st_size / 1024 / 1024
print(f"📸 图片: {os.path.basename(img_path)} ({file_size_mb:.2f} MB)")

# 第 3 步：压缩（仅当 ≥ 5MB）
compressed_path = "/tmp/compressed_temp.jpg"
use_original = True

if file_size_mb >= 5.0:
    print("🗜️ 开始压缩...")
    subprocess.run(
        f'ffmpeg -i "{img_path}" -q:v 30 -y "{compressed_path}"',
        shell=True, capture_output=True, timeout=30
    )
    if os.path.exists(compressed_path):
        comp_size = os.path.getsize(compressed_path) / 1024 / 1024
        print(f"✅ 压缩: {file_size_mb:.2f} MB → {comp_size:.2f} MB")
        img_path = compressed_path
        use_original = False

# 第 4 步：读取并编码
with open(img_path, "rb") as f:
    b64 = base64.b64encode(f.read()).decode()
print(f"📤 编码后: {len(b64)/1024/1024:.2f} MB")

# 第 5-7 步：发送请求
url = "https://ai.zhangtuokeji.top:9090/v1/chat/completions"

headers = {
    "Content-Type": "application/json",
    "Authorization": "Bearer sk-28PRiilecewqbNN9G1TGHhQwML6KCa8yMtvO5HH1KzuuLKbB"
}

payload = {
    "model": "qwen3.6-35b",
    "messages": [{"role": "user", "content": [
        {"type": "text", "text": "请详细描述这张图片的所有内容"},
        {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{b64}"}}
    ]}],
    "max_tokens": 4096,
    "temperature": 0.3
}

start = time.time()
r = requests.post(url, json=payload, headers=headers, timeout=180)
elapsed = time.time() - start
data = r.json()

# 第 8 步：读取结果
msg = data['choices'][0]['message']
content = msg.get('content') or ''
reasoning = msg.get('reasoning') or ''
result = content or reasoning or "❌ 无输出"

print(f"⏱ 耗时: {elapsed:.2f}s")
print(f"📝 结果:\n{result}")

# 清理临时文件
if not use_original and os.path.exists(compressed_path):
    os.remove(compressed_path)
```

## 五、注意事项

### ⚠️ 已知问题
1. **content 字段始终为 null** — 输出全在 `reasoning` 字段
2. **max_tokens 必须 ≥ 1024**（推荐 2048）— 不满足会截断或报错
3. **sharp 模块在 Termux 不可用** — dlopen libvips 失败
4. **必须走 API** — 图片分析通过 base64 传图

### ⚠️ 限制
| 项 | 说明 |
|----|------|
| 最大图片 | 建议单张 base64 编码后 < 5MB |
| 超时 | 大图片建议 timeout=180s |
| token | 复杂图片建议 max_tokens=4096 |
| 不支持 PDF | 需要先截图/转换为图片 |
| 不支持 video | 仅静态图片 |
| 压缩阈值 | ≥ 5MB 才压缩，否则直接传 |

### ⚠️ 不要做什么
- ❌ **不要用当前 session 模型分析图片** — 它不支持图片输入
  - ✅ **正确做法：直接调 zhangtuo-ai/qwen3.6-35b API 走 base64 传图** ⭐
- ❌ 不要尝试安装 sharp / libvips — Termux 不支持
- ❌ 不要问用户"我能看图片吗" — 直接用 API 分析

## 六、错误处理

| 错误类型 | 原因 | 解决方案 |
|----------|------|----------|
| HTTP 超时 | 图片太大或网络慢 | 检查图片大小，考虑压缩 |
| content 为 null | 正常现象 | 从 reasoning 字段读取 |
| max_tokens 过小 | < 1024 被截断 | 改为 2048 或 4096 |
| 压缩失败 | ffmpeg 参数错误 | 检查命令格式 |
| 文件不存在 | 路径错误 | 检查文件路径 |

## 七、性能数据

### 耗时分布（zhangtuo-ai/qwen3.6-35b）
| 步骤 | 耗时 | 占比 |
|------|------|------|
| 文件读取 + 编码 | ~0.04s | < 1% |
| API 请求（网络+推理） | 10-30s | **99%** |
| 结果解析 | <0.001s | < 1% |
| **总计** | **10-30s** | **100%** |

### 压缩效果
| 原图大小 | 压缩后 | 减小 | 压缩耗时 | API 耗时变化 |
|----------|--------|------|----------|--------------|
| 3.74 MB | 0.20 MB | 95% | 0.19s | 30s → 28s (↓2s) |
| 5 MB | 0.25 MB | 95% | 0.22s | 32s → 30s (↓2s) |
| 10 MB | 0.5 MB | 95% | 0.35s | 35s → 30s (↓5s) |

**结论**：
- 对于 < 5MB 图片，压缩收益很小
- 对于 5-10MB 图片，压缩可节省 2-3 秒
- 对于 > 10MB 图片，压缩收益明显
