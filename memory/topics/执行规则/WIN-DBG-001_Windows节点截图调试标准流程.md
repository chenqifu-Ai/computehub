# WIN-DBG-001 Windows 节点问题截图调试标准流程

> 版本: 1.0 | 制定: 2026-05-27

## 背景

Windows 节点（192.168.2.134）不能直接 SSH，也无法从 ECS 直接网络访问。调试的唯一通道是：
**老大在 Windows 上截图 → 发到群里/发给我 → 我分析 → 定位根因 → 修复**

## 标准流程

### 第 1 步：截图（老大操作）

在 Windows 节点上遇到 **任何报错、异常、崩溃、卡顿**，第一时间截图：

- 截全屏（Win+PrintScreen 或 微信 Alt+A）
- **不要只截错误弹窗**，尽量包含：标题栏、报错时间、完整的错误文本
- 如果错误日志很长，多截几张或拍完整

### 第 2 步：发送（老大操作）

截图发到群里或直接发给我，附上一句话说明：
- 当时在做什么操作
- 期望结果是什么
- 实际结果是什么

**不需要帮我分析，截了发就行**。

### 第 3 步：分析（小智）

我会：
1. 读截图上的错误信息和上下文
2. 查 Gateway 日志（ECS 上 gateway.log / worker.log）
3. 查节点注册状态（`/api/v1/nodes/list`）
4. 查 Worker 升级接口返回（`/api/v1/upgrade/check`）
5. 定位根因
6. **不要问问题，直接修**

### 第 4 步：修复（小智）

修复路径：
1. **代码问题** → 改 Go 源码 → 编译 → 推送到 ECS → 重启 Gateway
2. **配置问题** → 改 deploy 目录 / symlink / version.txt
3. **环境问题** → 写脚本走其他通道处理
4. 修复后更新本标准和 memory

## 当前已知的 Windows 调试限制

| 限制 | 说明 |
|------|------|
| 无法直接 SSH | 192.168.2.134 不通，ECS 也连不上 |
| 无法远程桌面 | 3389 端口没开或不对外 |
| Worker 日志不可达 | 日志在 Windows 本地，只有截图能看 |
| Worker 进程退出后不自动重启 | 需要老大手动启动 |

## 常见截图场景速查

| 截图标示 | 常见根因 | 处理方式 |
|----------|----------|----------|
| `connectex: actively refused` | Gateway 重启中 | 等几分钟自动恢复 |
| `connectex: No connection` | Gateway 挂了一段时间 | 检查 ECS 服务状态 |
| `平台不匹配: 期望 XXX` | 二进制命名不一致 | 改代码→编译→重启 Gateway |
| `404 Not Found` | deploy 目录缺少文件 | 建 symlink |
| worker 窗口消失 | 升级失败进程退出 | 修复后手动重启 worker |
