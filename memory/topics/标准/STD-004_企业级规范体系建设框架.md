## 📐 一、规范编号体系

```
┌─────────────────────────────────┐
│  前缀类型  │  编号范围       │  领域
├─────────────────────────────────┤
│  STD-XXX  │  001-099         │  通用运营标准
│  DEV-XXX  │  100-199         │  开发工程标准
│  OPS-XXX  │  200-299         │  运维操作标准
│  SEC-XXX  │  300-399         │  安全合规标准
│  QA-XXX   │  400-499         │  质量保证标准
│  DOC-XXX  │  500-599         │  文档管理规范
│  DATA-XXX │  600-699         │  数据管理规范
│  PROJ-XXX │  700-799         │  项目管理标准
│  HR-XXX   │  800-899         │  人力资源标准
│  FIN-XXX  │  900-999         │  财务运营标准
└─────────────────────────────────┘
```

### 编号格式

```
STD-XXX | 开发标准: DEV-XXX | 运维标准: OPS-XXX
```

- **XXX**: 三位数字，按时间顺序递增
- 前缀代表领域
- 编号一旦分配，永久不变

## 🔄 五、标准制定流程（SOP）

```
发现问题/需求
    ↓
分析根因（写进错误记录）
    ↓
制定方案（初稿）
    ↓
实战验证（至少1次真实场景）
    ↓
修订标准（基于验证结果）
    ↓
分配编号 + 写入标准目录
    ↓
通知相关方（邮件/消息）
    ↓
归档至 memory/topics/标准/
    ↓
更新 STD-004 标准索引
```

### 5.1 编号分配规则

| 触发场景 | 分配编号 |
|----------|---------|
| 修bug后提炼 | 下一个可用的 DEV/OPS/QA 编号 |
| 新领域首次定义 | 该领域下一个编号 |
| 现有标准修订 | 保留原编号，版本号+1 |
| 废弃标准 | 编号保留但不分配给新标准 |

### 5.2 状态流转

```
🟡 草案 ──验证──→ 🔵 评审 ──批准──→ ✅ 已生效
  ↑                                         │
  │                    ──修订──               │
  └─────────── 🔴 废弃（归档） ──────────────┘
```

## 🎯 七、标准执行机制

### 7.1 标准触发规则

```python
# 关键词触发（写入 STD-001 匹配引擎）
trigger_rules = {
    # 开发类
    "改代码"      → DEV-001 (代码审查),
    "版本发布"    → DEV-003 (版本管理),
    "合并分支"    → DEV-001 (代码审查),
    "写API"       → SEC-002 (API安全),
    
    # 运维类
    "部署"        → OPS-001 (启动规范),
    "安装软件"    → OPS-002 (安装标准),
    "改配置"      → STD-001 (标准匹配检查),
    "备份"        → DATA-001 (数据管理),
    
    # 通用类
    "发邮件"      → STD-003 (邮件发送),
    "改模型"      → DEV-004 (Provider合并),
    "删文件"      → SEC-001 (安全红线),
}
```

### 7.2 验证方式

| 标准 | 验证方式 | 频率 |
|------|---------|------|
| STD-001 | 修改配置时检查是否先读标准 | 每次修改 |
| STD-003 | 检查是否调用 `send_email.py` | 每次发邮件 |
| OPS-001 | 检查启动日志是否包含预期标志 | 每次部署 |
| OPS-002 | 检查安装后是否用完整路径验证 | 每次安装 |
