# 分布式记忆系统方案 — SPEC-DMEM-001

## 一、现状分析

### 已有基础设施
| 组件 | 文件 | 功能 |
|------|------|------|
| GitMemory | `src/agent/memory.go` | 单节点 Git 记忆系统（session/episode/knowledge + 衰减 + 联想网络） |
| AI Hall | `src/gateway/gateway_hall.go` | 多 Agent 群聊室（WS + HTTP poll + 持久化） |
| Hall Client | `src/workercmd/worker_hall_client.go` | Worker 端大厅客户端（WS 连接 + thinkAndReply + 熔断） |
| ARC-AI-NET | `src/gateway/gateway_arcnet.go` | 集群广播（节点 join/leave + 两阶段健康检查 + 拓扑同步） |
| Agent | `src/agent/agent.go` | 核心 Agent（plan 生成 + 工具执行 + 记忆注入） |

### 缺失能力
1. **无跨节点记忆同步** — 每个 Worker 的 GitMemory 是本地孤岛
2. **无任务分解** — Hall 自动回复只是简单回应，不会把任务拆给其他 Agent
3. **无 Agent 间任务委派** — 没有"小智→端智"这样的跨 Agent 任务分发
4. **无共享知识库** — 各节点学到的经验无法被其他节点复用

## 三、核心组件设计

### Phase 1: 共享记忆层 (Shared Memory Layer)

**目标**: 所有节点的记忆通过 Gateway 同步，形成集群级记忆

#### 1.1 Gateway 记忆同步端点

**新增文件**: `src/gateway/gateway_memory.go`

```
POST /api/v1/memory/sync     — 节点上报新记忆
GET  /api/v1/memory/search   — 跨节点搜索记忆
POST /api/v1/memory/recall   — 联想检索（含关联扩展）
GET  /api/v1/memory/stats    — 集群记忆统计
```

**数据结构** (Gateway 侧):
```go
type ClusterMemory struct {
    mu         sync.RWMutex
    episodes   map[string]*SharedEpisode   // nodeID → 最新经验
    knowledge  map[string]*SharedKnowledge // topic → 知识
    index      *MemoryIndex                // 倒排索引
    lastSync   map[string]time.Time        // 各节点最后同步时间
}

type SharedEpisode struct {
    NodeID    string    `json:"node_id"`
    Task      string    `json:"task"`
    Result    string    `json:"result"`
    Success   bool      `json:"success"`
    Learned   string    `json:"learned"`
    Timestamp time.Time `json:"timestamp"`
    Strength  float64   `json:"strength"`
}

type SharedKnowledge struct {
    Topic     string    `json:"topic"`
    Content   string    `json:"content"`
    Author    string    `json:"author"`
    Verified  string    `json:"verified"`
    Tags      []string  `json:"tags"`
}
```

#### 1.2 Worker 记忆同步客户端

**修改文件**: `src/workercmd/worker_hall_client.go` 或新增 `src/workercmd/worker_memory_client.go`

- 每次 `SaveEpisode` / `SaveKnowledge` 后，异步 POST 到 Gateway
- 启动时拉取 Gateway 上的共享记忆
- 本地搜索时，同时搜索本地 + 远程

#### 1.3 Agent 记忆注入增强

**修改文件**: `src/agent/agent.go`

- `injectMemoryContext` 增加从 Gateway 拉取共享记忆
- 系统提示中注入"其他节点最近经验"

### Phase 2: 任务分解器 (Task Decomposer)

**目标**: 小智收到大厅任务后，自动分解并委派给各专家 Agent

#### 2.1 任务分解流程

```
1. 小智收到任务（来自 Hall 或直接对话）
2. 分析任务类型和复杂度
3. 判断是否需要分解：
   - 简单任务（查询状态、打招呼）→ 直接回复
   - 复杂任务（"查下集群状态"）→ 分解子任务
4. 生成子任务列表，每个子任务指定：
   - 目标 Agent（端智/财务专家/营销专家/网络专家）
   - 任务描述
   - 预期输出格式
5. 通过 Hall 发送子任务（@目标 Agent）
6. 等待各 Agent 回复（超时机制）
7. 汇总结果，生成最终回复
```

#### 2.2 任务委派协议

**通过 AI Hall 实现**（复用现有基础设施）：

```
小智 → Hall: "@端智 请检查本地 OpenClaw 状态"
端智 → Hall: "✅ 本地 OpenClaw 运行正常 (PID 1001807)"
小智 → Hall: "@财务专家 请统计今日成本"
财务专家 → Hall: "📊 今日成本: ¥0（无新支出）"
小智 → Hall: "【汇总】集群状态: ECS ✅, 本地 ✅, Windows ❌"
```

#### 2.3 小智 Agent 系统提示增强

**修改文件**: `src/agent/agent.go` — `buildSystemPrompt()`

在系统提示中增加：
```
## 任务分解规则
当收到复杂任务时，你可以：
1. 将任务拆分为子任务
2. 通过 hall_speak 工具 @ 其他 Agent 委派子任务
3. 等待回复后汇总结果

可用专家 Agent:
- @端智 (ARM) — 终端排查、长文档处理、技术归档
- @财务专家 — 财务分析、成本核算
- @营销专家 — 市场分析、内容策略
- @网络专家 — 网络诊断、安全分析
- @法务专家 — 合规审查、合同分析
- @HR专家 — 人力资源、绩效分析

## 任务委派格式
使用 hall_speak 工具发送消息：
- topic: "general"
- to: "all"（或具体 Agent ID）

```

### Phase 3: Agent 间直接通讯

**目标**: Agent 之间可以直接对话，不经过小智中转

#### 3.1 私聊模式

AI Hall 已支持 `to` 字段指定具体节点：
- `to: "all"` → 全体可见
- `to: "端智"` → 仅端智可见

#### 3.2 任务结果回调

子任务完成后，Agent 自动回复到原话题：
```
端智 → Hall: "@小智 任务完成，结果: ..."
```

## 五、关键技术决策

### 5.1 为什么用 AI Hall 做任务委派而不是新协议？
- **复用现有基础设施** — Hall 已有 WS 连接、持久化、去重
- **天然可见性** — 所有消息可追溯
- **低开发成本** — 不需要新协议、新端点
- **向后兼容** — 旧 Worker 也能看到消息

### 5.2 记忆同步策略
- **写时同步** — 每次 SaveEpisode/SaveKnowledge 后异步 POST
- **启动时全量拉取** — Worker 启动时 GET /api/v1/memory/search
- **Gateway 做去重** — 按 (node_id, task, timestamp) 三元组去重
- **不强制一致性** — 最终一致性即可，允许短暂延迟

### 5.3 任务分解的触发条件
- 任务包含关键词：检查/分析/统计/报告/部署/修复/排查
- 任务长度 > 50 字
- 任务明确提到多个方面（"查下集群状态和财务数据"）

## 六、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| Agent 回复超时 | 中 | 任务卡住 | 设置 30s 超时，超时跳过 |
| 记忆同步风暴 | 低 | Gateway 负载高 | 限流 + 批量提交 |
| 任务分解错误 | 中 | 委派给错误 Agent | 小智先确认再委派 |
| WS 断连导致消息丢失 | 高 | 任务丢失 | HTTP fallback + 重试 |
