## 🔍 根因分析

### 直接原因

**Worker v1.3.35 内置 UpgradeManager 自动升级失败**，流程如下：

```
Worker 存活状态 (v1.3.35)
  ↓
Gateway: "有新版本 v1.3.38，SHA256=xxxx"
  ↓
UpgradeManager 决定执行升级
  ↓
下载新 binary (certutil -urlcache -f)
  ├── ⚠️ certutil 下载 >8MB 文件可能 hang/超时
  ├── ⚠️ URL 可能被 cmd 的 & 截断
  └── ⚠️ 旧 Worker 不知道自己下载到的是坏的
  ↓
旧 Worker 退出
  ↓
尝试启动新 binary
  ├── ⚠️ 新 binary 损坏 → 启动失败
  └── ⚠️ 或路径/参数不对 → 启动后连不上
  ↓
❌ 节点变砖：旧进程已死，新进程起不来
```

### 根本原因

```
┌─────────────────────────────────────────────────────┐
│ 升级即自杀：旧Worker先死，新Worker再跑              │
│                                                     │
│ 问题是——                                                   │
│ "如果新binary是坏的，谁负责验证？"                          │
│ 答案是：没人。旧Worker已经死了，没机会回滚。               │
└─────────────────────────────────────────────────────┘
```

### 间接原因

1. **IP 冲突驱逐** — `/home/computehub/worker.log` 中显示 wanlida-temp 和 phone-agent、windows-mobile 反复抢注册。IP 冲突（183.251.21.92 可能变了或共用出口）导致 Worker 注册不稳定
2. **5 小时 skip，凌晨突然触发** — 之前 50+ 次升级检查都 skip 了，为什么凌晨 3 点突然触发？Worker 的 upgrade_cache 冷却可能在此时过期
3. **无离线自愈机制** — Worker 挂了后没有服务/计划任务自动重启

## 🧠 事故教训（标准预防措施）

### 核心铁律

> **🚫 绝对禁止 Worker 内置自动升级（auto-upgrade）**
> Windows 节点的升级必须先下载→验证→确认版本对→再 kill 旧进程。
> 旧 Worker 不能自杀——它必须作为"守护者"活着，直到新 Worker 确认上线。

### 改进方案

#### 短期（立刻生效）

1. **关闭 wanlida-temp 的自动升级**（重新上线后设 `COMPUTEHUB_UPGRADE_STRATEGY=manual`）
2. **所有 Windows 节点升级走手动两步法**：
   - Step 1: 下载 + 验证 SHA256 + 确认版本 → 不 kill 旧进程
   - Step 2: 确认新 binary 没问题后，才 kill 旧进程 + 启动新的

#### 中期（代码层面）

1. **实现 test-register 模式**（v1.3.3 有过）：新进程先注册验证，确认连上 Gateway 了再通知旧进程退出
2. **增加升级回滚**：如果新进程 30s 内没上线，旧进程自动恢复
3. **Windows 节点自动重启机制**：注册为 Windows Service 或计划任务定时检查

#### 长期

1. Windows 节点的 C2 通道（WS/HTTP）之外，保留 SSH 或 RDP 兜底管理
2. Worker 升级状态上报到 Gateway，管理员可见"升级中/升级失败/回滚中"状态
