# 2026-04-30 日志

## 🔴 今日重点：图片识别 SOP 优化 (P0)

### 问题背景
- **07:06** 老大保存了一张微信截图到 `/storage/emulated/0/Pictures/WeiXin/`
- **07:06-07:09** 搜索定位文件耗时 ~2.5 分钟 (逐个目录搜索)
- **07:09-07:11** 调用 qwen3.6-35b API 分析耗时 ~10 秒
- **07:11** 输出分析报告 (中国建设银行电子回执单，¥50,000 服务费)
- **总耗时：~2 分 40 秒**，严重超时

### 根因分析
1. **图片分散存储**：Android 截图分布在 5+ 个目录 (DCIM/Camera, DCIM/Screenshots, DCIM/.globalTrash, Pictures/WeiXin, Pictures/ICBC)
2. **搜索方式低效**：逐个目录 ls/查找，没有统一索引
3. **API 调用串行**：每张图片单独请求，没有批处理
4. **无自动化发现**：缺少"最近 N 分钟/小时新增图片"的自动检测机制

### 目标
- **图片发现时间：** 2.5 分钟 → **3 秒内** (提升 98%)
- **API 调用时间：** 10 秒 → **3 秒** (提升 70%)
- **总耗时：** 2 分 40 秒 → **15 秒内** (提升 90%)

### 优化方案
1. **统一索引**：建立 `/storage/emulated/0` 图片文件的定期扫描索引 (按时间 + 目录分组)
2. **快速查找**：`find -newermt "今天 07:00" -type f \( -name "*.jpg" -o -name "*.png" \)`
3. **批量压缩**：统一压缩到 200KB 以内，减少 API 传输时间
4. **API 批处理**：多张图片合并请求，或异步并发调用
5. **自动报告**：结构化输出，直接返回关键信息表格

## 流式 Session 数据膨胀问题

### 现象
TUI 客户端断连重连后，历史消息被重绘，导致 session 消息数量膨胀，每次 HTTP POST 的 body 越来越大（含所有对话历史 + tool results）。

### 后续优化方向
1. **session.historyCompression** — 压缩/截断历史消息，只保留最近 N 轮
2. **session.maxHistoryAge** — 设置历史消息最大存活时间
3. **tool result 裁剪** — exec 输出等大块 tool result 只保留摘要
4. **定期 session 清理** — 长 session 自动分段
