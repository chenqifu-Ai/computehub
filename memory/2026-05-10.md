### 已确认的 6 个 Bug

1. **节点重复注册** — Worker 没有防二次注册机制，导致同一节点出现两次（一次空配置，一次有 region/GPU）。TUI 列表显示 5 行但只有 3 个真节点。
   - 触发文件: `cmd/worker/main.go` register() 无标志位
   - 影响: NodeManager.RegisterNode 虽然返回 already registered 错误但 Worker 不检查

2. **`completedcompleted` 显示 bug** — TUI 的 `visiblePad` 返回 `color+text+reset+spaces`，TUI 直接拼接 "completed" + visiblePad 结果，导致重复。
   - 触发文件: `cmd/tui/main.go` screenTasks() 中 `visiblePad(t.Status, sc, Reset, 10)`
   - 实际原因: visiblePad 已经加了颜色+重置，但外面又用 visiblePad 包了一层，或者 Status 本身包含 "completed" 两次拼接

3. **Exit code 127** — Android Termux 上 `hostname` 命令不存在（`echo WIN_OK && hostname` 执行到 hostname 时 exit 127）。不是 bug 是环境问题。

4. **调度器 dispatch() 是空函数** — `src/kernel/kernel.go` 中 `TaskDispatcher.dispatch()` 只有一行注释 `// Placeholder: dispatch pending tasks`。但老大发现任务提交走的是 `TaskSubmit.AssignedNode` 直接分配，不需要 dispatch()。

5. **任务提交必须指定节点** — 老大确认需要直接指定节点执行，不能靠自动调度。代码后端已经支持 `TaskSubmit.AssignedNode`（在 `kernel/actions.go` 的 `SubmitTask` 方法中），TUI 也实现了 `submitTask()` 发送 `assigned_node` 字段。所以后端已经 OK。

6. **cqf-worker-02 是 Windows 但装了 Linux Worker** — Worker 用 `sh -c` 执行，Windows 没 sh，exit 127。需要 Windows 版本的 Worker。

### 核心代码文件
- `src/kernel/actions.go` — TaskSubmit 结构体含 AssignedNode 字段，SubmitTask 支持直接指定节点
- `src/kernel/kernel.go` — ExtendedKernel 处理所有 action

- `src/gateway/gateway_worker.go` — Worker poll 接口，findPendingTaskForNode 按节点查找

- `cmd/tui/main.go` — TUI 界面
