## 2. 🧠 Worker Agent CEO 能力

每个 Worker 内置一个 Agent，能力和 Gateway 的 Agent 平级，但职责不同：

### 2.1 本地决策能力

| 场景 | 之前 (v1.0.2) | 现在 (v2) |
|------|------|------|
| 任务失败 | 回传 raw stderr | 本地分析根因 + 自动重试/降级 |
| 资源管理 | 被动等 Gateway 调度 | 自主判断"能不能接" + 主动汇报 |
| 断网 | 等死 | 本地正常运行，恢复后同步 |
| 多任务排序 | Gateway 决定 | Worker 自己按优先级排 |
| 本地文件处理 | 每个文件回传 | 本地过滤、预处理、压缩后再传 |
| 自我诊断 | 无 | 定期自检 CPU/内存/磁盘，主动预警 |

### 2.2 对话能力

Worker 需要有自己的对话端点，可以直接和它聊：

```
POST /api/v1/worker/think     # 直接和 Worker AI 对话
     {"task": "检查一下磁盘空间，如果不够就清理日志"}

POST /api/v1/worker/status    # 查看 Worker 状态（AI 理解版）
     {"query": "今天运行得怎么样？有没有异常？"}
```

### 2.3 LLM 策略

**不把 token 当成本**，但分场景优化：

| 场景 | 模型 | 位置 | 说明 |
|------|------|------|------|
| 重大决策 / 复杂分析 | qwen3.6-35b | Cloud (NewAPI) | 需要强推理 |
| 常规诊断 / 日常对话 | qwen3-coder-next | Cloud | 够用且更快 |
| 简单判断 / 紧急断网 | 内置规则引擎 | 本地 | 零延迟，不依赖网络 |

**断网降级策略：**
- 联网时: 走云端 LLM 做复杂决策
- 断网时: 走本地规则引擎，保证基础运转
- 恢复后: 自动同步断网期间的事件

## 3. 🔄 Layer 3: Brain 总裁办

Gateway 的 Layer 3 升级为"总裁办"，不再是唯一大脑：

### 3.1 职责变化

| 职责 | 说明 |
|------|------|
| 🧭 战略决策 | 跨节点任务分配、全局优先级、资源调配 |
| 📚 知识共享 | 全局记忆库 -> 各 Worker 可查询 |
| 🔗 协调仲裁 | 多个 Worker 争资源时做裁决 |
| 📊 全局监控 | 汇总各 Worker 状态，看全局健康度 |
| 🚀 战略部署 | 给 Worker 下发新能力、升级 Agent |

### 3.2 Gateway ↔ Worker 关系

```
不是主从关系，而是对等关系:

Gateway(总裁) ←→ Worker(子公司CEO)
    平等对话，各有各的AI大脑
    Gateway 有更大视野，做全局决策
    Worker 有自己的判断，不受微管理
```

## 4. 🚧 实施路径

### Phase 1 — Worker Agent 壳 (今明两天)
- Worker 内嵌 Agent 包（复用 `src/agent/` 的代码）
- Worker 注册 `/api/v1/worker/think` 端点
- Worker 本地工具: exec_shell, file_check, self_diagnose, retry_logic
- 支持本地简单规则引擎（断网降级）

### Phase 2 — Agent 对等通信 (本周)
- Gateway 的 Agent 可以调用 Worker 的 Agent
- Worker 之间可以互相通信（子公司间协作）
- 全局记忆 -> 分布式记忆（每个 Worker 有自己的记忆，可同步）

### Phase 3 — 全自主节点 (长期)
- Worker 能自主注册、自主升级、自主决策
- 新节点上线自动成为"智能体"，无需手动配置
- 多节点共识机制（选主、协调、容错）
