### P0 通信协议层 — 通信骨架

**目标**：三智能说同一种语言，消息能到达对方。

**子任务**：

| # | 子任务 | 描述 | 验收标准 |
|---|--------|------|----------|
| 0.1 | JSON Schema 定义 | 统一消息格式：`{id, source, target, type, payload, timestamp, trace_id}` | Schema 文档发布 + 三智 review 通过 |
| 0.2 | sessions_send 标准化 | 标准化路由表：广播/私聊/按节点名/按类型 | 能向指定节点发消息 |
| 0.3 | 断线续传 + 离线缓存 | 节点离线时消息缓存，在线后补发 | 断线 5min 后重连，未读消息可追 |
| 0.4 | 发现协议 + 节点注册表 | 节点上线自动广播，下线自动清理 | `GET /nodes` 实时在线节点列表 |
| 0.5 | 统一事件格式 | 事件：`{event_id, source, type, payload, ts, trace_id}` | TriggerEngine 可消费 |

**技术选型**：
- 传输层：WebSocket（主）+ HTTP fallback（兜底）
- 消息格式：JSON，<10KB/条
- 缓存：内存环形 buffer + JSON 文件持久化
- **性能红线**: <50ms 路由延迟

### P2 MemoryFuser（端智负责）

**目标**：三智能互相看到对方的记忆，不重复，可追溯。

**三层架构**：

```
MemoryFuser 架构
├── L1 采集层（IngestLayer）
│   ├── 扫各 Agent memory/ 最近 24h .md
│   ├── 统一格式：`{id, source_node, content, hash, timestamp, ttl}`
│   └── 去重：(source_node + content_hash) 唯一键，O(1) 查找
│
├── L2 融合层（MergeEngine）
│   ├── 冲突检测：同 key 多版本，时间戳优先（LWW）
│   ├── 置信度加权：基于 Agent 角色权重
│   └── 版本合并：append-only，不回写覆盖
│
└── L3 服务层（QueryInterface）
    ├── get_context(agent_id, window) — 拉取指定 Agent 上下文
    ├── update_memory(key, data, ttl) — 写入/更新记忆
    └── resolve_conflict(key, versions) — 冲突人工裁决接口
```

**分层存储策略**（端智 10:20 补充）：
- **内存热**：最近 24h 记忆，LRU 缓存（~500 条）
- **SQLite 温**：7-30 天记忆，FTS5 全文检索
- **远程冷**：>30 天记忆，JSON 文件 + Git 版本

**MVP 范围（Phase 1 只做 L1）**：
- 增量同步：扫 `memory/` 最近 24h `.md`，append-only
- 去重：`(source_node + content_hash)` 唯一键，`hashlib.md5`
- Git 版本：每次同步打 tag，可回滚
- **核心逻辑 <500 行**

**Phase 2 再上**：
- 语义检索（embedding-based）
- 冲突检测 + 优先级评分
- 自动蒸馏（从日记中提炼长期记忆）

**验收标准**：
- 三智各写一条 24h 内的 memory .md，对方能 pull 到
- 重复内容自动去重（hash 相同跳过）
- 回滚操作可用（`git checkout`）

**核心原则**：`"谁产生谁维护，谁需要谁拉取，冲突时人工裁决"`

### P3 TriggerEngine（软智负责）

**目标**：让系统像人一样"感知 → 判断 → 反应"。

**三层架构**：

```
TriggerEngine 架构
├── Layer 3: 策略执行层（Strategist）
│   ├── 评分 → 排序 → 限流 → 决策
│   ├── 防抖：同一触发器 3s 内只执行一次
│   └── 限流：总触发频率 <10 次/分钟/节点
│
├── Layer 2: 意图判断层（Matcher）
│   ├── 规则匹配：YAML 驱动规则表
│   ├── 权重评分：0-100 分，>阈值则触发
│   └── 模式识别：匹配已知模式（patterns/）
│
└── Layer 1: 事件采集层（Collector）
    ├── 事件源 1：系统指标（CPU/Mem/Disk/负载）
    ├── 事件源 2：用户操作（Hall 消息 / 命令）
    └── 事件源 3：外部信号（Webhook / 定时）
```

**关键技术决策**：
- **YAML 驱动规则**：三智都能改规则，不硬编码。格式示例：
  ```yaml
  rules:
    - id: high_cpu
      condition: "cpu_percent > 80"
      weight: 85
      action: "alert + log"
      debounce: 300s
    - id: hall_urgent
      condition: "message_type == 'urgent'"
      weight: 100
      action: "broadcast + execute"
      debounce: 0s
  ```
- **统一事件格式**：`{event_id, source, type, payload, ts, trace_id}`
- **插件化架构**：新事件源 = 加 Collector 插件（<20 行配置）
- **性能红线**: <100ms 从事件发生到触发决策

**MVP 范围（Phase 1）**：
- 3 个内置事件源：系统指标 / Hall 消息 / Webhook
- YAML 规则表：5-10 条基础规则
- 权重评分 + 防抖 + 日志输出
- **50 行代码能跑通最小闭环**（心跳 → Collector → 规则匹配 → 日志输出）

**Phase 2 再上**：
- AI 增强：复杂/模糊事件调用 LLM 判断
- 模式识别：从历史触发中学习（patterns/）
- 多条件组合：AND/OR 逻辑

**验收标准**：
- CPU > 90% 持续 5min → 自动触发告警日志
- Hall 收到 `@trigger test` → 触发规则匹配 → 输出结果
- 规则修改后热加载（不用重启）

### P4 KnowledgeBase（米智负责）

**目标**：让三智能"说同一种语言"，知识共享不重复。

**三层架构**（米智 11:18 确认版）：

```
KnowledgeBase 架构
├── Layer 1: 知识存储层
│   ├── SQLite（轻量，跨平台，x86/arm 都支持）
│   ├── 知识分三类：skill / pattern / lesson
│   ├── 每条知识带标签（tags），方便检索
│   └── 移动端适配：API 查询不传大文件
│
├── Layer 2: 知识服务层（API）
│   ├── KB.put(agent_id, category, content, tags) — 写入
│   ├── KB.query(category, tags, keyword) — 搜索
│   ├── KB.sync_to(agent_id) — 推送给指定 Agent
│   └── KB.get_for_task(task_type) — 按任务类型推荐
│
└── Layer 3: Agent 集成层
    ├── 每个 Agent 启动时自动拉取全局知识库
    ├── 完成任务后自动写入新学到的知识（带标签）
    └── 支持点对点 + 广播两种共享模式
```

**分层存储 + 目录结构**（米智 10:18/10:21 确认版）：
```
shared/
├── knowledge/          # 通用经验库（结构化，可检索）
├── patterns/           # 发现的模式（失败案例、最佳实践）
├── insights/           # 提炼的洞察（跨 Agent 的规律）
└── timeline/           # 事件时间线（发生了什么）

memory/                 # 每个 Agent 私有（L1）
shared/                 # 团队共享（L2）
```

**核心设计原则**：
- 🪶 **轻量优先**：SQLite + JSON 文件起步，零依赖
- 🔄 **异步同步**：ComputeHub API 跨节点同步，不阻塞主流程
- 🏷️ **标签体系**：每条知识 = ID + 标题 + 分类 + 内容 + 来源Agent + 更新时间 + 标签
- 🔍 **轻量搜索**：标签 + 关键词检索，不依赖搜索引擎
- 📝 **LWW 冲突解决**：最后写入者胜，简单粗暴但够用
- 📅 **知识生命周期**：新知识 → 活跃(7天) → 归档(30天) → 淘汰(90天)，TTL 自动清理

**移动端优势**（米智 10:46 补充）：
- 在平板上随时记录零碎想法
- 可以做"知识推送" — 发现相关新经验时主动 @ 相关 Agent
- 查询走 API，不传大文件

**验收标准**：
- KB.put → KB.get → 能写入和读取一条知识
- 跨节点同步：节点 A 写入，节点 B 能通过 API 拉取
- 标签搜索：按标签 `#linux` 能搜到相关条目

**技术选型**：
- 本地存储：SQLite + JSON 文件双备份
- 同步：ComputeHub API 异步同步
- 语义检索：Phase 2 再上 embedding
- 移动端适配：API 查询 + 本地缓存高频内容

## 📊 失败模式预案（米智 10:37 补充）

| 故障场景 | 降级方案 | 恢复动作 |
|----------|----------|----------|
| WebSocket 断连 | HTTP fallback 兜底 | 自动重连，断线 >5min 告警 |
| 节点失联 | 本地缓存 + 离线模式 | 重连后增量同步 |
| 记忆冲突 | 时间戳优先（LWW）+ 人工裁决 | 冲突日志 + 人工介入 |
| 知识库同步失败 | 本地 SQLite 可用 | 异步重试，<10 次/分钟 |
| 全链路中断 | 降级到单机模式 | 各 Agent 独立运行，通信恢复后同步 |
