## 一、目前的样子：骨架清晰，血肉稀疏

先说不坏的——底层架构其实不错：

- **ToolRegistry 设计**很有前瞻性，工具注册/描述/执行分离，加新工具只需几十行
- **Git Memory 系统**虽然是玩具版，但方向对，`recordEpisode`、`AppendSession`、`Recall` 这些接口都有了
- **SSE 流式支持**（`ThinkStream`/`ChatStream`）是加分项，前端体验不会死等
- **PlanStep 结构**有依赖关系、状态追踪、结果存储，流程控制的基础有了

但翻完代码后的真实感受是：**这个 Agent 现在约等于一个"能听懂人话的命令行脚本调度器"**。

## 三、跟它沟通的十大卡点（按严重程度排序）

### 🚨 #1 — 30 秒硬超时

```go
// agent.go:379 — execShell
maxWait := 30 * time.Second  // 写死的，不是从 step.Timeout 取的
```

任何运行超过 30 秒的任务都会返回 timeout。但 LLM 单次思考就要 5-10 秒，shell 任务跑起来更是几分钟打底。

**结果**: Agent 经常说"任务超时"，但你去看节点，任务其实跑完了。它不知道，也不会再问。

---

### 🚨 #2 — 不会上网（断网智能体）

没有 `web_search` 工具。你跟它说：
> "查一下 Go 1.24 发布了哪些新特性"
> "这个错误码是什么意思"

它只能凭训练数据的记忆回答——不稳定、不可靠、不新鲜。

---

### 🚨 #3 — 串行执行，卡一步全完

plan 是一步一步顺序执行的：

```
Step 1 (shell on ecs-p2ph) → 死等完成 → Step 2 → 死等完成 → Step 3
```

多节点并行？没有。Step 1 执行失败了后续步骤继续跑？是的——它不会停下来重新思考。

---

### 🚨 #4 — Plan 生成后永不修改

Agent 在最初的 LLM 调用中生成 plan，然后**严格按照这个 plan 执行到底**。即使 Step 1 的结果明显表明计划有问题，它也会硬着头皮执行完 Step 2、Step 3... 不会中途停下来重新调整。

---

### 🚨 #5 — 没有条件分支 / 循环 / 并行

PlanStep 的类型只有三种：
```go
case "shell":   // 执行命令
case "llm":     // 调大模型
case "tool":    // 调工具
```

没有 `"condition"`、`"parallel"`、`"loop"`。想表达"如果磁盘满了就清理，否则直接部署"——写不出来。

---

### 🚨 #6 — JSON 解析脆弱

```go
jsonStart := strings.Index(reply, "{")
jsonEnd := strings.LastIndex(jsonPart, "}")
```

纯字符串暴力截取。遇到：
- LLM 在代码块里输出 JSON（\`\`\`json ... \`\`\`）
- 嵌套 JSON
- 输出中带多余注释

这几个情况全部挂。

---

### 🚨 #7 — 记忆系统只写不读

Git Memory 确实在工作——每段对话、每次执行结果都记了。但 Agent **从来不主动利用这些记录**。`injectMemoryContext` 虽然调了，返回的只是简单的最近任务标题列表，没有语义理解、没有经验提取。

相当于一个人每天写日记，但从来不看。等于白写。

---

### 🚨 #8 — 失败不重试

Shell 执行失败 → 状态设为 `"failed"` → 汇报"出错了"。

不会做以下任何一件事：
- 换个节点重试
- 换一种命令
- 上网查错误码
- 自己修好再跑

---

### 🚨 #9 — 无主动意识 / 无心跳

它不会说：
> "老大，ecs-p2ph 已经离线 2 小时了"
> "磁盘快满了"
> "你上次让我跑的任务失败了，我已经换节点在重试了"

问它才答，不问装死。

---

### 🚨 #10 — 工具能力靠自己吹

虽然 ToolRegistry 能自动生成工具描述注入系统提示，但描述写得好不好、LLM 有没有看懂、记不记得用——全靠底模状态。没有人验证工具调用是否真正匹配。

## 四、升级路线图

### 🔧 Phase 0 — 先修 Bug（今晚就能干）

| 任务 | 改动 | 代码量 | 效果 |
|------|------|--------|------|
| 修复 30s 超时 | `maxWait = step.Timeout` 默认 300s | **3 行** | 再也不乱超时 |
| 优化 JSON 解析 | 先清代码块包裹，再用 json.Unmarshal | **10 行** | 提高 plan 成功率 |

### 🚀 Phase 1 — 工具扩容（2 天）

| 新工具 | 效果 | 代码量 |
|--------|------|--------|
| **web_search** | Agent 能联网查资料 | ~25 行 |
| **file_read** | 读日志/配置文件 | ~20 行 |
| **file_write** | 生成报告/代码文件 | ~20 行 |
| **git_commit** | 自动提交代码 | ~30 行 |
| **system_info** | 查节点 CPU/内存/磁盘 | ~20 行 |

加完这些，Agent 从"只会跑命令"变成"能跑能查能写"。

### 🧠 Phase 2 — 真正自主性（3-5 天）

1. **异步执行**：提交任务后不阻塞，同时调度多个节点 → 效率翻倍
2. **条件分支**：plan 支持 `if/then/else` → 自适应
3. **失败重试**：自动换节点/换命令 → 鲁棒性
4. **Plan 再生成**：执行中发现异常，重新 LLM 审视 → 灵活

### 🌟 Phase 3 — 学习进化（1-2 周）

1. **经验回放**：启动时回顾近期 failed episodes → 自动改良行为
2. **主动监控**：Agent 设置自己的心跳，定时检查集群健康 → 主动
3. **自改进**：从失败中提取规则，追加到自身 prompt → 越用越聪明

## 五、优先级建议

```
今晚上线 ──→ 修 30s 超时 + JSON 解析加强 ──→ 3 行代码，收益巨大
    ↓
这周吭哧 ──→ web_search + file_read + 条件分支 ──→ Agent 开始有"脑子"
    ↓
下周发力 ──→ 异步并行 + 失败重试 + 经验回放 ──→ Agent 开始有"智慧"
```

三个 Phase 干完，智能度预计能从 **35% → 70%+**。
