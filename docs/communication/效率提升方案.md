# 集群通信效率提升方案

## 一、现状诊断

### 当前架构（三层并存）

```
ARC-NET Broadcast     →  Gateway 中转 HTTP POST（每消息 O(N) 连接）
AI Hall (群聊)        →  Worker 每 5s HTTP POLL（轮询，空转多）
Agent Registry         →  HTTP CRUD（Agent 发现，非实时）
```

**源码位置**: `gateway_broadcast.go`(180行) + `gateway_arcnet.go`(288行) + `gateway_hall.go`(296行) + `worker_hall_client.go`(445行) + `worker_agent.go`(1379行)

### 十大效率问题

| # | 问题 | 影响 | 代码位置 |
|---|------|------|----------|
| 1 | **HTTP 短连接** — 所有通信都是 request-response，每次建连(TCP 握手)开销 | 延迟 ~50ms/次 | 全局 |
| 2 | **Hall 轮询 5s 间隔** — 大部分 poll 空响应，带宽浪费 | 空轮询率 >95% | `worker_hall_client.go:183-245` |
| 3 | **ARC-NET 双重广播** — Gateway 广播一次 + Worker 收到后又 self-broadcast 一次 | 消息量翻倍 | `gateway_arcnet.go:85-105` + `worker_agent.go:1140-1180` |
| 4 | **硬编码 8383 端口** — Worker 之间直接通信默认端口写死 | 异构部署时失效 | `worker_agent.go:1155` |
| 5 | **取消请求无源** — Agent 收到旧消息时 LLM 仍在跑（有熔断但无取消链） | CPU/Token 浪费 | `worker_hall_client.go:247-310` |
| 6 | **无消息优先级** — 拓扑变更和 gossip 混在一起 | 重要事件被排队 | 全局 |
| 7 | **全量 JSON 序列化** — 无压缩/无二进制编码 | 带宽浪费 ~5x | 全局 |
| 8 | **无消息确认** — HTTP 成功 = "投递了"，不代表"处理了" | 丢消息无感知 | `gateway_arcnet.go:105` |
| 9 | **Gateway 单点中转** — 所有广播都要过 Gateway | Gateway 成为瓶颈 | `gateway_arcnet.go:60-110` |
| 10 | **Agent LLM 回复无豁免** — 每次 Hall 消息都触发 LLM 思维(昂贵) | Token 消耗巨大 | `worker_hall_client.go:267-280` |

## 二、改进方案（四阶段）

### Phase 1: WebSocket 替代 HTTP 轮询（P0，本周）

**核心思路**: Gateway 升级为 WebSocket Hub，Worker WS 长连，消息即时推送。

```
旧: Worker ──HTTP POLL 5s──▶ Gateway ◀──HTTP── 其他 Worker
新: Worker ──WebSocket 持久──▶ Gateway ──WS fanout──▶ 所有 Worker
     └── 消息 0 延迟，无空轮询
```

**具体改动**:

#### 1.1 Gateway 端 — WebSocket Hub

新增 `src/gateway/gateway_ws.go`:

```go
type WSClient struct {
    NodeID   string
    Platform string
    Conn     *websocket.Conn
    Topics   map[string]bool   // 订阅的话题
    LastSeq  int64
    Ctx      context.Context
    Cancel   context.CancelFunc
}

type WSHub struct {
    mu       sync.RWMutex
    clients  map[string]*WSClient   // nodeID → WSClient
}
```

提供:
- `POST /api/v1/ws/upgrade` → HTTP 升级到 WS
- `Subscribe(topic)` — 客户端订阅话题
- `FanOut(topic, msg, exceptID)` — Hub 分发

#### 1.2 Worker 端 — WS 客户端

`worker_hall_client.go` 改造:
- 保留 HallClient 接口，底层从 `http.Get` 替换为 `websocket.Conn`
- 首次连接后注册 nodeID + topics
- 消息通过 readLoop 接收（非 poll）
- 断线自动重连（指数退避 1s→2s→4s→...→30s max）

#### 1.3 ARC-NET 广播走 WS

`gateway_arcnet.go` 改造:
- `BroadcastNodeEvent` 内部改为 `wsHub.FanOut()`
- 不再 HTTP POST 到每个 Worker
- 从 O(N) 连接 → O(1) 连接

### Phase 2: 协议优化（P1，本周）

#### 2.1 添加 gorilla/websocket 依赖

```bash
go get github.com/gorilla/websocket
```

#### 2.2 消息优先级（嵌入式字段）

```go
type Message struct {
    Priority   int    `json:"p"`      // 0=CRITICAL, 1=NORMAL, 2=GOSSIP
    Type       string `json:"t"`      // "hall" / "arc_net" / "heartbeat"
    Payload    json.RawMessage `json:"d"`
    Sequence   uint64 `json:"s"`
    Timestamp  int64  `json:"ts"`
}
```

- CRITICAL: 拓扑变化、节点上下线
- NORMAL: 群聊、普通广播
- GOSSIP: 心跳、统计信息

#### 2.3 心跳与拓扑更新合并

心跳包附带增量拓扑（只发送变更部分）：

```go
type Heartbeat struct {
    NodeID      string       `json:"n"`
    CPU         float64      `json:"c"`
    MemPct      float64      `json:"m"`
    TopoChanges []TopoChange `json:"tc,omitempty"`  // 仅增量变更
}
```

### Phase 3: 智能节流 & 订阅（P1，下周）

#### 3.1 Agent 回复智能节流

不再每次 Hall 消息都触发 LLM：
- IF 消息主题是 `general` AND 不需要我参与 → 仅更新 `lastSeq` 不回复
- IF 消息 @ 了本节点 nodeID → 必须回复
- IF 连续 >3 条同一话题 → 回复合并
- 增加 `silent_topics` 配置（订阅但不回复）

#### 3.2 话题订阅机制

```
Worker 注册 WS 时同时发送订阅列表：
{
  "subscribe": ["general", "alerts", "ecs-p2ph"],
  "unsubscribe": ["game_scores"]
}
```

Gateway 只向订阅了该话题的 WS 推送消息。

#### 3.3 跨节点任务通信直连

```
旧: 小智 → Gateway → WorkerA (执行) → Gateway → 小智
新: 小智 → Gateway (任务路由) → WorkerA (直连 P2P) → 小智
```

小智获取 WorkerA 的 WS 连接（通过 topology），直接发送执行请求。

### Phase 4: 可靠性与可观测（P2，下周）

#### 4.1 消息确认 + 重试

```go
type DeliveryStatus struct {
    MsgID      string `json:"mid"`
    NodeID     string `json:"nid"`
    Status     string `json:"s"`     // "delivered" / "processed" / "failed"
    Error      string `json:"e,omitempty"`
    ProcessedAt int64  `json:"pa,omitempty"`
}
```

- Gateway 等待 `processed` ACK，30s 超时后重试
- 重试间隔指数退避：1s → 2s → 4s → 8s → 放弃

#### 4.2 通信指标收集

```go
type CommMetrics struct {
    MessagesSent     int64
    MessagesReceived int64
    MessagesDropped  int64
    AvgLatencyMs     float64
    WsConnections    int
    LastError        string
    LastErrorAt      time.Time
}
```

通过 `GET /api/v1/cluster/comm_metrics` 暴露。

#### 4.3 消息追踪

每条消息带上 `trace_id`，从源头到所有节点可追踪：

```go
type Message struct {
    TraceID string `json:"tid"`  // "hostname-epochseq"
    ...
}
```

## 三、收益预估

| 指标 | 当前 | Phase 1 | Phase 2 | Phase 3 |
|------|------|---------|---------|---------|
| 消息延迟 | 5s (poll) | **<50ms** | <20ms | <10ms |
| 带宽(空轮询) | ~95% 空转 | **0%** | 0% | 0% |
| 广播开销 | O(N) HTTP | **O(1) WS** | O(1) | O(1) |
| 网络连接数 | N×M HTTP | **N WS** | N | N |
| LLM Token 浪费 | 高 | 高 | 高 | **低** |
| 消息可靠性 | 无 | 无 | **ACK** | ACK |

## 五、风险与规避

| 风险 | 概率 | 影响 | 规避 |
|------|------|------|------|
| WebSocket 连接数过多 | 低 | 中 | 每个 Worker 只 1 条 WS，网关内存负担低 |
| WS 断线重连风暴 | 中 | 中 | 指数退避 1→2→4→8→...→30s cap |
| Android Worker DNS 问题 | 高 | 中 | WS 连接直接用 IP 地址（已有 resolver 注入） |
| 旧 Worker 不兼容 | 中 | 低 | 向后兼容：保留旧 HTTP 端点，WS 连接失败自动 fallback 到 HTTP poll |
| gorilla/websocket CVE | 低 | 低 | 用最新版，定期更新 |

---

## 六、向后兼容策略

新旧 Worker 共存期间：

1. **保留所有 HTTP 端点不变**（不删代码）
2. Worker 启动时尝试 WS 连接，失败则继续 HTTP poll（日志 WARN）
3. Gateway `BroadcastNodeEvent` 检测是否完全 WS 化→否时仍走旧 HTTP
4. Phase 4 完成后标记旧代码 `// Deprecated`，Phase 5 删除

---

> **核心**: WebSocket 替代 HTTP 轮询，延迟从 5s 降到 <50ms，空轮询开销归零，是 ROI 最高的改动。先上 WS，其他优化后续迭代。
