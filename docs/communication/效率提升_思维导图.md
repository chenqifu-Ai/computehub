```mermaid
mindmap
  root((集群通信效率提升))
    
    🔴 现状痛点
      HTTP短连接
        每次建TCP握手的开销
        延迟叠加 ~50ms/次
      Hall 5秒轮询
        空轮询率 >95%
        大量带宽浪费
      ARC-NET双重广播
        Gateway发一次
        Worker收到又发一次
        消息量翻倍
      Agent盲目回复
        每条Hall消息都调LLM
        Token消耗巨大
      可靠性缺失
        无消息确认ACK
        失败不重试
        丢消息无感知
      硬编码8383端口
        异构部署失效
      无消息优先级
        拓扑变更和八卦混一起
    
    --- Phase 1 ---
    🟢 WebSocket替换轮询
      Gateway新增WS Hub
        Upgrade HTTP→WebSocket
        管理所有Worker长连接
        FanOut分发消息
      Worker改WS客户端
        替换5s HTTP poll
        ReadLoop接收消息
        断线自动重连
        指数退避 1s→30s
      ARC-NET改走WS
        BroadcastNodeEvent→WS FanOut
        从O(N)降到O(1)
        删除旧HTTP广播
      核心收益
        延迟: 5s → <50ms
        空轮询: 95% → 0%
        连接数: N×M → N条WS
    
    --- Phase 2 ---
    🟢 协议优化
      消息优先级
        p=0 CRITICAL: 拓扑变更
        p=1 NORMAL: 群聊广播
        p=2 GOSSIP: 心跳统计
      心跳+拓扑合并
        心跳包附带增量拓扑
        只发送变更部分
      取消双重广播
        砍掉worker_agent.go sendBroadcast中的self-broadcast
        只由Gateway统一分发
      端口动态获取
        从topology API获取节点端口
        不再硬编码8383
    
    --- Phase 3 ---
    🟢 智能节流&订阅
      Agent回复节流
        @本节点才回复
        连续同话题→合并回复
        增加silent_topics配置
      话题订阅机制
        Worker注册时提交订阅列表
        Gateway按话题推送
        不订阅的不推送
      跨节点P2P直连
        任务路由走Gateway
        执行结果直连返回
        减少中转跳数
    
    --- Phase 4 ---
    🟢 可靠性与可观测
      消息确认ACK
        Gateway等待processed确认
        30s超时重试
        指数退避1s→2s→4s→放弃
      通信指标收集
        GET /api/v1/cluster/comm_metrics
        消息数/延迟/丢包/连接数
      消息追踪
        每条消息带trace_id
        全链路可溯源
    
    📊 收益预估
      延迟
        当前: 5s
        Phase1: <50ms
        Phase2: <20ms
      带宽
        当前: 95%空转
        优化后: 0%空转
      广播开销
        当前: O(N) HTTP
        优化后: O(1) WS
      Token浪费
        当前: 高(每条消息调LLM)
        优化后: 低(智能节流)
      可靠性
        当前: 无确认
        优化后: ACK+重试
    
    📅 实施计划
      Day1 Phase1核心
        go get gorilla/websocket
        gateway/ws.go WSHub
        WS升级端点注册
      Day2 Phase1完成
        worker WS客户端
        断线重连
        ARC-NET改WS
      Day3 Phase2协议
        消息优先级字段
        心跳拓扑合并
        砍掉双重广播
      Day4 Phase3节流
        Agent回复规则
        话题订阅机制
      Day5 Phase4可靠
        ACK+重试
        指标收集
        全链路回归
    
    ⚠️ 风险规避
      WS断线重连风暴
        规避: 指数退避+cap
      Android DNS问题
        规避: WS直连IP
      旧Worker不兼容
        规避: 保留HTTP端点做fallback
      gorilla/websocket CVE
        规避: 用最新版
```
