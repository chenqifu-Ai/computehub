# 🚀 ComputeHub 工业级落地工作计划书 (v2.0)

**项目目标**：构建分布式算力出海平台 v2.0 (工业级实现)
**执行周期**：12 周
**核心原则**：代码先行，物理驱动，状态确定。

---

## 🟢 第一阶段：物理原点构建 (Week 1 - 3)
**核心目标**：实现 Node <<->-> Gateway 的真实物理连接。

| 周次 | 工作重点 | 核心交付物 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **W1** | **物理心跳实现** | `node/monitor.py` | 能实时采集 GPU 温度、显存、功耗并生成硬件指纹 |
| **W2** | **确定性网关搭建** | `api/rest_api.py` | 基于 FastAPI 实现节点注册和心跳接收 API |
| **W3** | **物理资源库建设** | `db/schema.sql` | PostgreSQL 存储节点物理指纹，实现心跳时序记录 |

---

## 🟡 第二阶段：确定性调度引擎 (Week 4 - 6)
**核心目标**：实现任务在分布式网络中的“无模糊”状态转移。

| 周次 | 工作重点 | 核心交付物 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **W4** | **任务状态机开发** | `scheduler/fsm.py` | 实现 Submitted -> Executing -> Completed 的严格转移 |
| **W5** | **防御性调度算法** | `scheduler/engine.py` | 实现基于物理延迟的 L3 匹配 + 区域熔断机制 |
| **W6** | **任务分发链路** | `node/executor.py` | 实现从网关接收任务 -> 本地调度执行 -> 结果回传的闭环 |

---

## 🟠 第三阶段：物理真实验证层 (Week 7 - 9)
**目标**：用物理事实取代数字信任，构建计费闭环。

| 周次 | 工作重点 | 核心交付物 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **W7** | **双节点冗余验证** | `validation/verifier.py` | 关键任务双跑，通过 Hash 比对判定结果真实性 |
| **W8** | **真实算力计费** | `blockchain/settlement.py` | 基于 GPU 实际利用率时长计费，剔除空转时间 |
| **W9** | **区块链结算闭环** | `blockchain/contracts.sol` | 部署智能合约，实现 Verifier 签名 -> Token 释放 |

---

## 🔴 第四阶段：极致性能与可视化 (Week 10 - 12)
**目标**：将系统推向生产级性能，实现全球算力可见。

| 周次 | 工作重点 | 核心交付物 | 验收标准 |
| :--- | :--- | :--- | :--- |
| **W10** | **gRPC 极速通道** | `api/grpc_server.py` | 将心跳和分发链路从 HTTP 迁移至 gRPC，延迟降低 50% |
| **W11** | **全球算力地图** | `web/dashboard.html` | 实现基于地理位置的节点分布热力图和流量可视化 |
| **W12** | **SDK 封装与压测** | `sdk/python_client.py` | 完成 Python SDK，执行 1000+ 虚拟节点压力测试 |

---

## 🛠️ 资源与风险管控

### 1. 关键路径依赖
- **硬件依赖**：需要至少 2 台带有 NVIDIA GPU 的机器用于物理验证。
- **网络依赖**：需要跨地域的服务器来测试区域熔断和延迟调度。

### 2. 风险预警
- **驱动兼容性**：不同 CUDA 版本可能导致 monitor.py 崩溃 -> 对策：建立兼容性矩阵。
- **区块链延迟**：主网结算慢 -> 对策：采用 Layer 2 或先在测试网运行。
