问题1：配置文件格式错误

错误日志

[02:57:30] ERROR Config invalid File: ~/.openclaw/openclaw.json Problem: - gateway.bind: gateway.bind host aliases (for example 0.0.0.0/localhost) are legacy; use bind modes (lan/loopback/custom/tailnet/auto) instead (auto-migrated on load). Gateway aborted: config is invalid. Fix the config and retry, or run "openclaw doctor" to repair.

问题分析

根本原因
  • gateway.bind使用了旧格式（如 "0.0.0.0" 或 "localhost"）
  • 新版本要求使用bind modes：lan/loopback/custom/tailnet/auto
  • 配置文件编辑时没有正确更新格式

解决方案

# 方法1：运行自动修复 openclaw doctor --fix # 方法2：手动修改配置 # 旧格式（错误） "gateway": { "bind": "0.0.0.0" } # 新格式（正确） "gateway": { "bind": "lan" }

问题2：Gateway重启超时

错误日志

[02:58:29] WARN config change requires gateway restart deferring until 2 operation(s), 1 embedded run(s) complete [02:59:59] WARN restart timeout after 90210ms with 2 operation(s), 1 active embedded run(s) still active; restarting anyway [02:59:59] received SIGUSR1; restarting draining 2 active task(s) and 1 active embedded run(s) before restart

问题分析

发生了什么
  • 配置更改后需要重启Gateway
  • 但还有活跃任务在运行（2个任务+1个嵌入运行）
  • 等待90秒后强制重启
  • 这可能导致数据丢失或任务中断

解决方案

# 修改配置前，先确保没有活跃任务 openclaw gateway status # 如果有活跃任务，等待完成后再修改 # 或者手动重启 openclaw gateway restart

问题3：LLM请求超时

错误日志

[05:40:34] WARN embedded_run_agent_end "error":"LLM request timed out." "failoverReason":"timeout" "model":"glm-5:cloud" "provider":"ollama" [05:40:46] WARN embedded_run_agent_end "error":"Ollama API error 400" {"error":"prompt too long; exceeded max context length by 3 tokens"}

问题分析

两个相关问题
  • 超时：模型请求时间过长，可能是网络问题或模型负载高
  • 上下文溢出：prompt太长，超出模型上下文窗口限制
  • 这两个问题可能同时发生

解决方案

# 解决超时：检查网络连接 curl -s http://192.168.1.7:11434/api/version # 解决上下文溢出：压缩会话历史 # 会话太长时，考虑重启会话或压缩历史 # 切换到更大的模型 vim ~/.openclaw/openclaw.json # 使用context window更大的模型

问题4：Gateway启动失败 - 端口占用

错误日志

[05:49:18] ERROR Gateway failed to start gateway already running (pid 28837); lock timeout after 5000ms If the gateway is supervised, stop it with: openclaw gateway stop [05:49:19] ERROR Port 18789 is already in use. - Port is in use but process details are unavailable (install lsof or run as an admin user). [06:01:12] ERROR Gateway failed to start gateway already running (pid 28837); lock timeout after 5000ms [06:01:18] ERROR Gateway install failed Error: systemctl not available; systemd user services are required on Linux.

问题分析

发生了什么
  • Gateway进程已在运行（PID 28837）
  • 端口18789被占用
  • 尝试启动新实例失败
  • Termux环境没有systemd

解决方案

# 方法1：停止现有Gateway openclaw gateway stop # 方法2：强制杀死进程 kill -9 28837 # 方法3：检查端口占用 netstat -tlnp | grep 18789 ss -tlnp | grep 18789 # Termux环境：使用前台模式运行 # 不要用systemd，直接前台运行

问题5：systemd不可用

错误日志

[05:54:06] ERROR systemd user services unavailable. systemd user services are unavailable; install/enable systemd or run the gateway under your supervisor. If you're in a container, run the gateway in the foreground instead of `openclaw gateway`. [06:01:18] ERROR Gateway install failed Error: systemctl not available; systemd user services are required on Linux.

问题分析

环境限制
  • OpenClaw默认使用systemd管理Gateway服务
  • 但Termux/Android环境没有systemd
  • 这导致无法安装为系统服务

解决方案

# Termux环境下，使用前台模式运行 # 不要尝试用systemd # 方法1：直接运行Gateway openclaw gateway start # 方法2：前台模式（推荐） openclaw # 这会在前台运行，适合Termux # 检查Gateway状态 openclaw gateway status
✅ 注意事项
Termux环境下，Gateway会在前台运行。如果关闭终端，Gateway也会停止。

问题6：模型上下文窗口太小

错误日志

[06:15:18] ERROR blocked model (context window too small) ollama/glm-5:cloud ctx=8192 (min=16000) source=model [06:15:18] ERROR lane task error "error":"FailoverError: Model context window too small (8192 tokens). Minimum is 16000." [06:15:18] ERROR Embedded agent failed before reply Model context window too small (8192 tokens). Minimum is 16000.

问题分析

核心问题
  • 模型 glm-5:cloud 配置的上下文窗口是8192
  • 但系统要求最小16000 tokens
  • 这是模型配置问题，不是会话问题
  • 连续多次失败，任务无法执行

解决方案

# 方案1：修改模型配置，增加context window vim ~/.openclaw/openclaw.json # 在models配置中设置更大的context window "models": { "providers": { "ollama": { "models": [ { "id": "glm-5:cloud", "contextWindow": 127000 } ] } } } # 方案2：切换到其他模型 "primary": "ollama/deepseek-coder:latest"

问题7：MEMORY.md文件缺失

错误日志

[02:21:44] ERROR [tools] read failed: ENOENT no such file or directory, access '/root/.openclaw/workspace/MEMORY.md' [02:45:22] ERROR [tools] read failed: ENOENT no such file or directory, access '/root/.openclaw/workspace/MEMORY.md'

问题分析

发生了什么
  • 系统启动时尝试读取MEMORY.md
  • 但文件不存在
  • 导致记忆系统无法正常工作

解决方案

# 创建MEMORY.md文件 touch ~/.openclaw/workspace/MEMORY.md # 添加基础内容 echo "# MEMORY.md - Long Term Memory" > ~/.openclaw/workspace/MEMORY.md # 确保memory目录存在 mkdir -p ~/.openclaw/workspace/memory
✅ 最佳实践
每次会话结束后，系统会自动更新memory/日期.md文件。MEMORY.md用于长期记忆。

问题8：配置编辑无变化

错误日志

[06:10:29] ERROR [tools] edit failed No changes made to /root/.openclaw/openclaw.json. The replacement produced identical content. This might indicate an issue with special characters or the text not existing as expected.

问题分析

可能原因
  • 查找的文本不存在（大小写、空格、特殊字符问题）
  • 替换的内容和原内容相同
  • 文件已被修改，找不到原始文本

解决方案

# 方法1：先查看当前配置 cat ~/.openclaw/openclaw.json | head -50 # 方法2：使用精确匹配（注意大小写、空格） # 确保oldText完全匹配文件中的内容 # 方法3：直接写入新配置 vim ~/.openclaw/openclaw.json # 或使用write工具完全重写

开发教训

⚠️ 关键教训
  • 配置修改要谨慎：修改前备份，修改后验证
  • 了解环境限制：Termux没有systemd，使用前台模式
  • 监控资源使用：上下文窗口限制会导致任务失败
  • 日志是朋友：/tmp/openclaw/openclaw-*.log 包含所有错误
  • 等待任务完成：不要在任务运行时强制重启

常用诊断流程

# 1. 检查Gateway状态 openclaw status # 2. 查看错误日志 grep ERROR /tmp/openclaw/openclaw-$(date +%Y-%m-%d).log | tail -20 # 3. 检查配置 cat ~/.openclaw/openclaw.json # 4. 运行诊断 openclaw doctor # 5. 如有需要，修复并重启 openclaw doctor --fix && openclaw gateway restart
