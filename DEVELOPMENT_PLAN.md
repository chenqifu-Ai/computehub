## 🎯 融合策略：继承 + 升级 + 补全

### 三大原则

1. **继承 OpenPC 的工程规范** — go.sum, 测试, 脚本, 二进制构建全部补齐
2. **升级 ComputeHub 的业务模型** — GPU 任务 + 状态机 + 配额，不回归 OpenPC 字符串模式
3. **补齐分布式和区块链** — 这是 ComputeHub 区别于 OpenPC 的核心价值

## 🔧 技术栈决策

| 层次 | 技术 | 理由 | 来源 |
|------|------|------|------|
| 语言 | **Go 1.24+** | 并发好、编译快、部署简单 | OpenPC 继承 |
| TUI | **tcell + tview** | 终端 UI 标准库 | OpenPC 继承 |
| Web | **HTMX + Go templates** | 极简全栈 | 新项目 |
| 框架 | **net/http + chi** | 轻量高性能 | OpenPC 继承 |
| 数据库 | **SQLite** → **PostgreSQL** | 开发友好 → 生产扩展 | 渐进升级 |
| 缓存 | **Redis** | 任务队列、节点缓存 | 新项目 |
| 容器化 | **Docker** | 沙盒隔离 | 阶段一新增 |
| 节点通信 | **gRPC** | 高性能远程调用 | 阶段二新增 |
| 区块链 | **Solidity + web3.go** | 合约 + Go 交互 | 阶段三新增 |
| 监控 | **Prometheus + Grafana** | 行业标准 | 阶段四新增 |

## 🏆 执行标准 (融合版)

遵循 SOUL.md 六大原则 + 三大支柱:

1. **绝对确定性** — 内核严格状态机 + 分布式最终一致性
2. **防御性鲁棒性** — 节点故障自动熔断 + 优雅降级
3. **物理真实交付** — 计费基于真实 GPU 利用率
4. **多级纯化** — 任务进入内核前四层过滤
5. **环境解耦** — 能力协商，梯度可用
6. **可回溯进化** — 一次出错，永久修复 (基因系统)

## 🔄 开发流程

```
每日站会 → 代码审查 → 测试覆盖 → 自动部署 → 监控验证
   ↓
 Sprint 回顾 → 经验总结 → 基因进化 (更新 genes.json)
```

### 代码质量标准
- 测试覆盖率 ≥ 80%
- 所有 PR 必须通过 CI

### 发布标准
- 所有测试通过
- 性能基准达标 (核心响应 < 50ms)
