## 1. 执行摘要

### 一句话定位

**ComputeHub** 是一个开源的分布式算力出海平台，通过去中心化网络连接全球闲置 GPU 与 AI 算力需求方，提供成本仅为云服务 30-50% 的高性能算力交易市场。

### 核心数据

| 指标 | 数值 |
|------|------|
| 代码库 | **~11,300 行 Go** 代码 |
| 测试覆盖 | **194 个单元测试**，8/8 包全绿 |
| 开发阶段 | **Sprint 1-4 完成**（执行引擎→分布式→区块链→Web界面） |
| 核心架构 | 确定性内核 + 4级纯化 + 基因进化 + 6策略调度 |
| 计费模式 | 5种物理计费（时间/GPU利用率/Token/推理次数/混合） |
| 结算方式 | 区块链智能合约自动结算 |
| 技术栈 | Go 1.24+ / 无外部依赖 / 轻量部署 |

### 愿景

> **让每个人的闲置 GPU 都能赚钱，让每个人都用得起 AI 算力。**
>
> — *"Start local. Scale globally."*

### 市场机会

全球 GPU 算力市场在 2026 年已达到 **$500 亿+**，年增长率 35%+。但：
- 云 GPU 价格高昂（A100 ~$3-4/小时）
- 全球大量 GPU 闲置（游戏、挖矿、数据中心空闲节点）
- 一人公司/独立开发者被价格门槛挡在 AI 门外

**ComputeHub 的目标**：建立一个算力"Airbnb"——让算力供给端和需求端直接匹配，去掉云厂商 50-70% 的中间商溢价。

## 2. 问题与解决方案

### 核心问题

#### 🔴 算力供给侧：大量 GPU 闲置浪费

| 场景 | 闲置率 | 原因 |
|------|--------|------|
| 个人游戏显卡 (RTX 4090) | ~80% | 每天玩 4 小时，闲置 20 小时 |
| 矿卡退役 (RTX 3080/3090) | ~90% | ETH 转 POS 后大量闲置 |
| 数据中心空闲节点 | ~30-40% | 业务波峰波谷差异 |
| 高校/研究机构 | ~60% | 寒暑假、非高峰时段 |

**全球闲置 GPU 算力估值**：相当于 300-500 万张 A100 等效算力（保守估计）

#### 🔴 算力需求侧：价格门槛阻碍创新

| 场景 | 云服务月费 | 年成本 | 承受能力 |
|------|-----------|--------|----------|
| 独立开发者做模型微调 | $3,000+ | $36,000 | ❌ 几乎不可承受 |
| 小团队推理服务 | $5,000+ | $60,000 | ❌ 吃掉了大部分利润 |
| 学生/研究者实验 | $500-1,000 | $6,000-12,000 | ❌ 负担沉重 |
| 一人AI公司起步 | $2,000+ | $24,000 | ❌ 烧光启动资金 |

#### 🔴 信任问题：算力交易缺乏可信中介

- 算力提供方：怕跑完了收不到钱
- 算力使用方：怕付了钱跑不出结果
- 现有方案：云服务商中心化抽成 50-70%

### 解决方案

#### ✅ ComputeHub 三层解耦架构

```
┌──────────────────────────────────────────────────┐
│ 信任层: 区块链智能合约自动结算 (无需信任中介)      │
├──────────────────────────────────────────────────┤
│ 调度层: 确定性内核 + 智能调度 (任务可靠执行)       │
├──────────────────────────────────────────────────┤
│ 执行层: 沙盒隔离 + 物理验证 (算力真实交付)         │
└──────────────────────────────────────────────────┘
```

#### 具体价值

| 角色 | 问题 | ComputeHub 方案 |
|------|------|-----------------|
| **算力提供方** | GPU 闲置无用 | 连接网络 → 自动接任务 → 赚 CHB Token |
| **算力需求方** | 云 GPU 太贵 | 按需租用 → 成本降 50-70% → 真实物理计费 |
| **双方共同** | 缺乏信任 | 智能合约托管 + 自动结算 → 无需信任 |
| **平台运营** | 冷启动难 | 开源社区驱动 → 先跑通私有场景 → 逐步扩展 |

**三大工程差异化优势**：

| 优势 | 描述 | 竞争对手做不到的 |
|------|------|-----------------|
| **绝对确定性** | 严格状态机 + 线性队列，无竞态 | 大多用异步 + 消息队列，状态不可预测 |
| **物理真实计费** | 基于 GPU 利用率/Token 消耗，非简单时长 | 大多数按时间来，对需求方不公平 |
| **防御性鲁棒性** | 区域熔断 + 自动恢复，局部故障不级联 | 很多平台节点一崩就全崩 |

### 4.3 完整结算链路（已实现）

```
客户 充值CHB → 登记任务 → 创建托管锁定资金
               ↓
       节点执行任务（GPU采样每分钟）
               ↓
       任务完成 → 物理计费引擎计算
               ↓
       释放托管：付给节点 + 退款给客户
               ↓
       全部记录上链 → 可追溯、不可篡改
```

### 7.3 差异化优势总结

1. **物理计费引擎** — 不是简单地按时长计价，而是基于真实 GPU 利用率、Token 消耗、推理次数。这是目前竞品都没有做到的。
2. **确定性内核** — 从 OpenPC 继承的严格状态机 + 线性任务队列，保证任务执行的可预测性。这在分布式算力场景下至关重要。
3. **梯度降级沙盒** — 从 Docker → cgroup → chroot → shell 的梯度隔离，让没有 Docker 环境的节点也能参与。
4. **基因进化系统** — 失败模式自动学习，一次出错永久修复。这是算力网络长期稳定运行的关键能力。
5. **纯 Go + 无外部依赖** — 二进制部署，单文件运行，部署成本趋近于零。

### 7.4 竞品不足 & 我们的机会

| 竞品 | 不足 | ComputeHub 机会 |
|------|------|-----------------|
| **AWS Spot** | 随时中断、价格波动大、锁定厂商 | "永不中断"的确定性任务 + 成本-50% |

## 12. 风险与应对

### 12.1 风险矩阵

| 风险 | 概率 | 影响 | 等级 | 应对策略 |
|------|:----:|:----:|:----:|----------|
| **冷启动失败** | 🔴 高 | 🔴 高 | 🚨 | 先跑通私有场景，用真实数据说服 |
| **GPU 提供方不足** | 🟡 中 | 🔴 高 | ⚠️ | 集中攻矿工社群 + 高校实验室 |
| **需求方不足** | 🟡 中 | 🔴 高 | ⚠️ | 先提供免费额度，积累案例 |
| **区块链 gas 费过高** | 🟡 中 | 🟡 中 | ⚠️ | 分层结算 (链下聚合 + 链上最终结算) |
| **合规风险 (跨境算力)** | 🟡 中 | 🔴 高 | ⚠️ | 先做国内/单一市场，合规后再出海 |
| **竞品先发优势** | 🟡 中 | 🟡 中 | ⚠️ | 差异化: 物理计费 + 确定性内核 |
| **技术债务/安全漏洞** | 🟢 低 | 🔴 高 | ⚠️ | 代码审查 + 安全测试 + Bug Bounty |
| **团队单一** | 🔴 高 | 🔴 中 | ⚠️ | 尽早引入合作者，降低单点风险 |
| **市场价格战** | 🟡 中 | 🟡 中 | ⚠️ | 开源社区护城河 + Token 经济锁定 |

### 12.2 关键风险应对策略

#### 🚨 冷启动失败 — 最高风险

**核心问题**: 算力市场是典型的双边网络，供给和需求必须同时存在。

**应对方案**:
1. **不追求一步到位** — 先做私有算力池，自己提供设备，跑通完整链路
2. **单边启动** — 先集中做供给方（GPU 提供者），用免费额度吸引需求方
3. **垂直场景切入** — 不泛做"通用算力"，先聚焦"LLM推理"一个高频场景
4. **开源社区口碑** — 以技术影响力吸引早期用户，而非烧钱营销

#### ⚠️ 合规风险 — 影响最大

**核心问题**: 算力跨境交易涉及各国监管，处理不当可能导致法律风险。

**应对方案**:
1. **阶段化出海** — 先做国内市场或单一司法管辖区
2. **合规优先** — 正式运营前完成 GDPR 基础合规
3. **去中心化治理** — 通过 DAO 分散法律风险
4. **法律顾问** — 与专业律师合作，建立合规框架
